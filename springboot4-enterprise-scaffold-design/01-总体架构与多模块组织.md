## 2. 仓库 / 模块结构（推荐方案 C：脚手架 mono-repo + 业务独立仓）

### 2.1 脚手架仓库（本仓库）

```
springboot4-enterprise-scaffold/
├── pom.xml                       # 父 POM（仅做聚合 + 插件管理）
├── bom/                          # 唯一允许定义版本号的模块
│   └── pom.xml                   #   所有依赖版本（含本地化包）
├── shared-kernel/                # 共享内核（业务方不可绕过）
│   ├── kernel-core/              #   错误码、统一上下文、Trace、雪花 ID
│   ├── kernel-exception/         #   异常体系 + 全局兜底
│   ├── kernel-logging/           #   日志规范 + 脱敏
│   ├── kernel-web/               #   统一返回体、全局 Advice、参数校验
│   └── kernel-test/              #   测试基类、Testcontainers 抽象
├── starters/                     # 能力模块（按需引入）
│   ├── starter-datasource/       #   多数据源 + 公共库驱动路由
│   ├── starter-cache/            #   Redis / Caffeine 二级缓存抽象
│   ├── starter-messaging/        #   消息中间件统一抽象
│   ├── starter-ai/               #   Spring AI 封装 + RAG pipeline
│   ├── starter-auth/             #   认证鉴权 + 三方 IdP
│   ├── starter-config/           #   配置中心客户端 + 总线监听
│   ├── starter-gateway-sdk/      #   被网关回调的 SDK（健康、版本上报）
│   └── starter-observability/    #   日志/指标/链路一体化
├── tools/                        # 工具类二次封装（详见 08）
│   ├── tools-lang/               #   字符串/集合/日期/反射封装
│   ├── tools-crypto/
│   └── tools-http/
├── local-libs/                   # 超期包本地化（详见 08）
│   └── README.md                 #   每个包的"为什么本地化 + 替换计划"
├── archetypes/                   # Maven Archetype（业务方一键生成骨架）
│   └── biz-app-archetype/
├── examples/                     # 示例业务服务（最小可运行）
│   └── example-biz-app/
└── build/                        # CI/CD 模板、ArchUnit 规则、Checkstyle 配置
    ├── ci-templates/
    ├── archunit-rules/
    └── checkstyle/
```

### 2.2 业务仓库（业务团队 fork 或基于 archetype 生成）

```
biz-app-xxx/
├── pom.xml                       # 继承脚手架 BOM
├── app-api/                      # 对外接口契约（OpenAPI YAML + 生成的 DTO）
├── app-domain/                   # 领域层（Entity、Domain Service，无框架依赖）
├── app-infra/                    # 基础设施实现（Repo、外部调用）
├── app-application/              # 应用编排层（Use Case）
├── app-web/                      # Web 层（Controller、Advice）
└── app-bootstrap/                # 启动模块（main + 配置）
```

> **关键约束**：业务仓内的层间依赖只允许 `web → application → domain ← infra`，由 ArchUnit 强制。

## 3. 子项目之间的交互方式（"接口式"如何落地）

### 3.1 三种合法跨子项目交互模式

| 模式 | 适用 | 实现 | 缺点 |
|------|------|------|------|
| **OpenAPI（同步）** | 跨服务查询 / 命令 | Gateway 路由 + 生成 SDK Jar 给调用方 | 强依赖被调方在线 |
| **消息契约（异步）** | 解耦的业务事件 | 共享 `event-contract` 模块，仅含 Avro/Proto schema | 排查链路较长 |
| **共享只读视图** | 跨域报表 / 大数据 | 物化视图 / CDC 到独立读库 | 不实时 |

> **禁止**：直接 Maven 依赖另一个 BizApp 的 Service 类、共享数据库表写权限、跨服务 JDBC 读取对方业务表。

### 3.2 SPI 边界示例（脚手架内 starter 之间也遵循）

- `starter-messaging` 暴露 `MessageProducer`、`MessageConsumer` 接口
- 具体实现 `messaging-kafka`、`messaging-rocketmq` 作为子模块按需启用
- 业务方只 import 接口模块，运行时通过条件装配选择实现

## 4. 同一项目内部的组织方式（你特别问的"同项目里怎么办"）

### 4.1 推荐：DDD 轻量分层 + Use Case 编排

```
app-web         （Controller, 仅做参数转换 + 调用 Use Case）
   ↓
app-application（UseCase / CommandHandler，事务边界在此）
   ↓
app-domain     （Entity, ValueObject, DomainService，无 Spring 依赖）
   ↑
app-infra      （Repository 实现, 外部接口调用, 适配器）
```

### 4.2 同项目内"模块化"的两种粒度

| 粒度 | 何时用 | 落地 |
|------|--------|------|
| **Maven 子模块** | 业务域 ≥ 3 个、团队 ≥ 5 人 | 每个域一个 module，编译期隔离 |
| **包级模块（package-by-feature）** | 中小项目 | `com.xxx.feature.order / feature.user`，ArchUnit 限制跨包调用 |

### 4.3 强制规则（ArchUnit）

- `feature.A` **禁止**直接调用 `feature.B.internal`，只能调用 `feature.B.api`
- `domain` 层**禁止**依赖 `org.springframework.*`
- `web` 层**禁止**直接依赖 `infra` 层

## 5. 共享内核（Shared Kernel）边界

### 5.1 哪些东西必须放共享内核（业务不可绕过）

- 全局异常体系 + 错误码注册表
- 统一返回体 `Result<T>` 与序列化规则
- TraceContext / 用户上下文 / 租户上下文
- 雪花 ID / UUID v7 生成器
- 日志脱敏注解与切面
- 时间 / 货币 / 金额等基础值对象

### 5.2 哪些不放共享内核（避免"上帝模块"）

- 业务实体、业务规则
- 第三方 SDK 适配（放对应 starter）
- 任何带状态的服务

## 6. 业务团队接入流程（脚手架易用性的试金石）

```
1. mvn archetype:generate -DarchetypeArtifactId=biz-app-archetype
2. 修改 application.yml 中：app.tenant、app.bizdb-pool
3. 启动：默认连 local 环境的 CommonDB、Nacos、Redis
4. 写第一个 UseCase：参考 examples/example-biz-app
5. CI 自动套用脚手架的 ArchUnit + Checkstyle + Sonar 规则
```

## 8. 待澄清问题

1. 业务方是否会有"非 Java 服务"接入？若有，跨语言契约必须走 Proto/Avro，会影响 D1.4。
2. 业务方对"领域驱动"的接受度？若团队 Java 经验薄，DDD 四层可能成本过高，可降级为经典三层 + Use Case。
3. 是否需要支持"同一脚手架同时输出 Web + Worker + Job"三种形态？影响 archetype 数量。
