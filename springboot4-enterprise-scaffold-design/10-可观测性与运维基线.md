## 2. 日志

### 2.1 输出
- 结构化 JSON（开发环境保留人类可读）
- 通过 stdout 输出，由 sidecar / DaemonSet（Promtail / Filebeat）采集
- 字段规范见 09 章

### 2.2 关键链路标记
- 所有请求入口注入 traceId（OTel 自动）
- 业务关键节点 `log.info("event=order.created", "orderId", id)`
- 慢操作 / 异常 / 兜底 必须打日志

### 2.3 留存策略
| 类型 | 留存 |
|------|------|
| INFO 及以上 | 30 天热 + 180 天冷 |
| ERROR | 1 年 |
| 审计日志（鉴权 / 配置变更 / AI 调用） | 6 月 ~ 3 年（按合规） |

## 3. 指标

### 3.1 必采指标（脚手架自动暴露）

| 类别 | 指标 |
|------|------|
| **HTTP** | RPS、P50/P95/P99、错误率、状态码分布 |
| **JVM** | 堆 / 非堆 / GC 次数 / 线程数 |
| **DB** | 连接池使用率、慢查询数、SQL P99 |
| **缓存** | hit ratio、qps、本地+远程拆分 |
| **消息** | 生产 RPS、消费滞后、DLQ 计数 |
| **AI** | 调用次数 / token / 成本 / 失败率（按模型 / 任务 / 租户） |
| **业务** | 业务方自定义（@Counted / @Timed） |

### 3.2 标签规范
- 必带：`app, env, version, instance, tenant_id?`
- 高基数标签禁止（如 user_id），改用日志聚合
- 由脚手架统一注册 `MeterFilter`，业务方违规自动报警

## 4. 链路追踪

- 自动埋点：HTTP / JDBC / Redis / Kafka / RestClient / Feign / Spring AI
- 跨服务透传：W3C Trace Context（标准 `traceparent` header）
- 异步上下文：脚手架 TaskDecorator 强制透传 SpanContext
- 采样策略：
  - test/local：100%
  - beta：50%
  - pro：尾采样 + 错误必留 + 业务关键 100%
- 与日志关联：日志中 traceId 与 trace 系统一致，可一键跳转

## 5. 健康检查与就绪检查（K8s 必备）

| 探针 | 用途 | 行为 |
|------|------|------|
| **Liveness** | 进程是否活着 | 失败 → 重启 Pod |
| **Readiness** | 是否能接流量 | 失败 → 从 Service 摘除（不重启） |
| **Startup** | 启动期长任务 | 启动期间禁用 Liveness |

**脚手架默认实现**：
- Liveness：仅检查 Spring 上下文 + JVM 健康
- Readiness：依赖检查（DB / Nacos / Kafka 连通），任一失败 → not ready
- Startup：启动 90s 内（可调），完成数据源 / 缓存 warmup

**优雅下线**：
- preStop hook 调用 `/actuator/shutdown` → 标记 not ready → 等 LB 摘除（≥ 15s）→ 拒绝新请求 → 等待在飞请求结束（最多 60s）→ 退出
- 配合 K8s `terminationGracePeriodSeconds: 60`

## 6. SLO / SLI 基线（建议默认值）

| 服务类型 | SLO 目标 |
|---------|----------|
| 核心交易 API | 可用性 99.9% / P99 < 500ms |
| 普通业务 API | 99.5% / P99 < 1s |
| AI 同步对话 | 99% / P99 首 token < 2s |
| AI 异步任务 | 95% 5min 内完成 |
| 消息消费滞后 | P95 < 30s |

> SLO 未达成自动触发 PagerDuty / IM 告警 + 错误预算消耗仪表盘。

## 7. 告警分级

| 级别 | 触发 | 通知 | 响应 SLA |
|------|------|------|---------|
| P0 | 核心交易不可用 / 数据库宕 | 电话 + IM | 5 min |
| P1 | SLO 显著偏离 / 错误率突增 | IM @ 值班 | 15 min |
| P2 | 单次异常 / 容量预警 | IM | 工作时间 |
| P3 | 趋势预警 | 邮件 | 周内 |

**告警治理**：
- 禁止 ALERT_FATIGUE：每条告警必须可处置，否则降级或删除
- 季度评审告警有效率（命中率 / 误报率）

## 8. 灰度与回滚（与 02、03 衔接）

### 8.1 自动回滚条件（Argo Analysis）
- 错误率 5 分钟超 1%
- P99 延迟超基线 30%
- 关键业务指标（订单成功率）下降超 5%

### 8.2 回滚动作
1. Argo Rollouts 自动 rollback 到前一版本
2. 同时把 Nacos 路由权重切回旧版
3. 告警通知 + 自动开 Postmortem 模板
4. 阻塞同分支后续合并直到根因分析完成

### 8.3 应急止血开关（Kill Switch）
- 配置中心保留每个核心模块的 enabled 开关
- 灾难时一键关闭某能力（不重启服务）

---

## 9. 容量与压测

- 每个服务必须有压测基线（脚手架提供 Gatling / k6 模板）
- 关键服务每月一次全链路压测（在 beta 用影子库 / 流量染色）
- 容量评估：CPU 60% 即扩容，避免突发打挂
- HPA 基于自定义指标（不仅 CPU），如消息消费滞后
