# 08 · 依赖治理与工具封装

> 解决：**只用近 4 年发布的包、超期包如何"放本地慢慢淘汰"、流水线如何检查、工具类二次封装方便替换、BOM 统一管理**

---

## 1. 总体策略


## 2. "近 4 年"策略的精确定义

| 状态 | 判定 | 处理 |
|------|------|------|
| **健康** | 最近一次正式版发布 ≤ 3 年 | 正常使用 |
| **预警** | 3 年 < 最近发布 ≤ 4 年 | CI 警告 + 寻找替代物登记 |
| **超期** | 最近发布 > 4 年 | 禁止从中央仓拉取，必须降级到 local-lib |
| **豁免** | 行业标准 / 无替代（如 ASM、AspectJ） | 显式登记到 `bom-exemptions.yaml` |

**"最近发布"判定数据源**（按优先级）：
1. Maven Central API（`search.maven.org`）
2. OSS Index（同时检查漏洞）
3. 自建镜像仓的发布时间元数据

## 3. 你提到的"超期包放公共 lib，打包时本地引入"——落地方案

### 3.1 目录约定

```
local-libs/
├── README.md                         # 全局说明（怎么管、怎么查）
├── group-id-x/
│   ├── artifact-y/
│   │   ├── 1.2.3/
│   │   │   ├── artifact-y-1.2.3.jar
│   │   │   ├── artifact-y-1.2.3.pom
│   │   │   ├── artifact-y-1.2.3-sources.jar
│   │   │   └── META.yaml             # 来源、原始仓库、登记理由、替换计划
└── ...
```

### 3.2 引入方式（两选一，**强烈推荐方案 B**）

#### 方案 A：`<scope>system</scope>` + `systemPath`
```xml
<dependency>
  <groupId>com.example</groupId>
  <artifactId>legacy-x</artifactId>
  <version>1.2.3</version>
  <scope>system</scope>
  <systemPath>${project.basedir}/local-libs/.../legacy-x-1.2.3.jar</systemPath>
</dependency>
```
- 优点：零基础设施
- 缺点：传递依赖坏死、IDE 难索引、`mvn package` 不打入 jar

#### 方案 B：私有 Maven 仓库 + local-libs 自动同步（**推荐**）
```
local-libs/  ──► CI（pre-merge）──►  Nexus/Harbor/Artifactory 私服
                                            │
                                            ▼
                            业务 pom 通过正常 <dependency> 引用
                            （路径与公网坐标一致，但仅私服有）
```
- 优点：与正常依赖体验一致、传递依赖正常、IDE 友好
- 缺点：需要私服

> 流水线检查"只检查 maven 拉取"——方案 B 完美匹配你的诉求：拉取来自私服，公网检查不报警。

#### 方案 C（混合）
- 默认私服镜像（拉取优先私服，私服无则透传公网）
- 超期包先放私服 `local-libs-repo`，主仓库代理屏蔽公网同名 GAV

## 4. 包替换的"缓慢淘汰"机制

每个 local-lib 必须配套 `META.yaml`：

```yaml
groupId: com.example
artifactId: legacy-x
version: 1.2.3
imported_at: 2026-04-15
last_release: 2020-08-10
reason: |
  原作者归档，无安全修复但功能稳定。
  暂无完美替代，作为过渡。
replacement_plan:
  candidates:
    - { name: modern-y, status: 评估中, eta: 2026-Q3 }
  blockers:
    - modern-y 缺少 zigzag 解析器
review_at: 2026-Q3
owner: arch-team@xxx
```

**自动机制**：
- 季度任务扫描 `META.yaml`，过期未 review 自动开 issue
- 引入新依赖前，CI 检查替代候选是否已可用，若可用则报红
- 仪表盘公示：local-libs 数量、最长在库时长、归还进度

## 5. 工具类二次封装（你强调的"方便替换"）

### 5.1 原则
- **第三方工具类（Hutool / Guava / Commons-* / Apache HttpClient / OkHttp）禁止业务代码直接 import**
- 必须经 `tools-*` 模块包装
- 包装层暴露的是"语义接口"，不是"工具厂商接口"

### 5.2 模块划分

| 模块 | 包装的内容 | 内部实现 |
|------|-----------|----------|
| `tools-lang` | 字符串 / 集合 / 反射 / 日期 / 数字 | 默认 Apache Commons + JDK 21 新 API |
| `tools-crypto` | 加解密 / 签名 / 哈希 / 国密 | BouncyCastle |
| `tools-http` | HTTP 客户端 | JDK HttpClient（默认）/ OkHttp（备选） |
| `tools-json` | JSON 序列化 | Jackson（默认）/ Fastjson2（备选） |
| `tools-id` | 雪花 / UUID v7 / 短码 | 自研 |
| `tools-time` | 时区 / 周期 / 计时 | java.time + 封装 |
| `tools-file` | 文件流 / 压缩 / 上传 | NIO + Apache Compress |
| `tools-validate` | 业务校验组合（手机/邮箱/身份证） | 自研 + Hibernate Validator |

### 5.3 替换底层的方式
- 内部用 SPI / 条件装配
- 替换底层时业务代码零改动
- 例：`HttpExecutor` 接口默认 JDK 实现，加个依赖即可切到 OkHttp 实现

### 5.4 命名约束
- 包装类名不带厂商名（`StringHelper` 而不是 `HutoolStringUtil`）
- 包路径统一 `com.xxx.tools.{domain}`
- 所有公开方法必带 javadoc + 单元测试

## 6. BOM 模块（依赖唯一定义点）

```
bom/pom.xml
├── <dependencyManagement>
│   ├── Spring Boot 4.x BOM        ← 引官方
│   ├── Spring Cloud 4.x BOM
│   ├── Spring AI BOM
│   ├── 我们自己的 starters         ← 自管版本
│   ├── 第三方依赖（healthy 状态）   ← 自管版本，统一升级
│   └── local-libs（系列）          ← 与私服 GAV 一致
```

**强制规则**：
- 业务 pom 只能 `<scope>import</scope>` BOM
- 业务 pom 写 `<version>` 标签 → CI 拒绝
- 升级依赖只在 BOM 这一处改，所有业务方 mvn 重新构建即生效

## 7. 漏洞与许可证治理

| 维度 | 工具 | 卡点 |
|------|------|------|
| **CVE** | OWASP Dependency-Check / Trivy | 高危直接 fail，中危预警 |
| **License** | License Maven Plugin | GPL / AGPL 禁止商用项目，黑白名单 |
| **传递依赖** | mvn dependency:tree CI 输出 | diff 报警异常引入 |
| **冗余依赖** | maven-dependency-plugin analyze | 提示移除 |

## 10. 待澄清问题

1. 是否已有私有 Maven 仓库？没有的话需要先做基础设施。
2. 公司是否有许可证黑白名单？否则需要先建立合规规范。
3. 现存项目是否大量直接用 Hutool / Fastjson 等？影响迁移到 tools-* 的工作量。
4. "4 年"是否对所有依赖一刀切？JDK / Spring Boot 本身？官方 LTS 是否豁免？
   - 我的建议：**官方根 BOM（Spring Boot / JDK / Cloud / AI）按 LTS 政策走，不受 4 年限制；其他第三方 4 年硬卡。**
