# 04 · 多数据源与持久化

> 解决：**1 公共库 + N 业务库怎么动态切换、读写分离、分库分表预留、向量库 / RAG 库的位置**

---

## 1. 数据源拓扑

```
                ┌──────────────────────────────────┐
                │          应用启动                  │
                │   1. 连接 CommonDB (公共库)        │
                │   2. 读取 ds_route 表             │
                │   3. 创建 N 个业务数据源（懒加载） │
                └──────────────┬───────────────────┘
                               │
        ┌──────────────────────┼──────────────────────────┐
        │                      │                          │
        ▼                      ▼                          ▼
┌──────────────┐      ┌──────────────────┐       ┌──────────────┐
│  CommonDB    │      │  BizDB[1..N]     │       │ Vector DB     │
│  ─ tenants   │      │  - 业务真实数据   │       │ - pgvector /  │
│  - ds_route  │      │  - 按租户/业务分库 │       │   Milvus      │
│  - app_meta  │      │  - 主从读写分离   │       │ - RAG 索引    │
│  - audit     │      │                  │       │               │
└──────────────┘      └──────────────────┘       └──────────────┘
```

## 2. 公共库（CommonDB）的角色

**定位**：所有 BizApp **唯一共享**的元数据库，**只存路由 / 元数据 / 跨业务字典**，不存任何业务实体。

### 2.1 核心表（示意，非最终 DDL）

| 表名 | 作用 |
|------|------|
| `tenant` | 租户元数据 |
| `ds_route` | 数据源路由：`tenant_id / biz_code → ds_key` |
| `ds_pool` | 数据源连接信息：`ds_key → jdbc_url, secret_ref, pool_config` |
| `app_registry` | 业务应用注册：`app_name → 允许使用哪些 ds_key` |
| `feature_flag` | 全局特性开关 |
| `audit_log` | 跨服务审计 |
| `i18n_dict` | 国际化字典 |
| `region_meta` | 地域元数据（多活场景） |

### 2.2 启动加载逻辑（业务无感知）

1. `starter-datasource` 从 application.yml 读 CommonDB 连接
2. 启动时拉 `ds_route` + `ds_pool`，构建 `Map<dsKey, DataSource>`
3. 注册一个 `RoutingDataSource`（继承 Spring `AbstractRoutingDataSource`）
4. 业务方注解 `@DataSource(ds = "biz_xxx")` 或在请求上下文注入 `tenantId`，路由由切面决定
5. 监听 CommonDB 的 `ds_route` 变更事件（基于配置中心或 DB CDC），动态增删数据源，**不重启**

## 3. 数据源切换的三种触发模式（建议三种共存）

| 模式 | 适用 | 实现 |
|------|------|------|
| **注解显式** | 跨库定时任务、管理后台 | `@DataSource("biz_07")` AOP 切面 |
| **租户隐式** | SaaS 业务接口 | 网关注入 `X-Tenant-Id` → ThreadLocal → 路由 |
| **业务码隐式** | 同租户多业务域 | URL/参数解析 `bizCode` → 路由 |

**ThreadLocal 注意事项**：
- 必须在 `@RestControllerAdvice` 入口设置、出口清理
- 异步线程（`@Async` / CompletableFuture）必须用 TaskDecorator 透传
- 响应式（WebFlux）走 Reactor Context，不能用 ThreadLocal

## 4. 读写分离

### 4.1 推荐：ShardingSphere 5.x（业内主流，和 Spring Boot 4.x 兼容）

- 主从配置在 CommonDB.ds_pool 中按 `master / slaves[]` 维护
- 读路由策略：`ROUND_ROBIN / WEIGHTED / LEAST_LATENCY`
- 强一致读用 `@MasterOnly` 注解强制走主库

### 4.2 备选：自研轻量路由

仅在 starter-datasource 内做主从切换，不引入 ShardingSphere 全套（避免复杂度）。

| 方案 | 优点 | 缺点 |
|------|------|------|
| ShardingSphere | 功能全（读写分离 + 分库分表 + 加密 + 影子库） | 包大、学习成本高 |
| 自研 | 轻、可控 | 后续要分库分表时要重写 |

**推荐**：默认引 ShardingSphere，业务方用不到的特性不开。

---

## 5. 分库分表（预留方案，不强制）

- **不在 v1 启用**，但 starter-datasource 设计要兼容
- 当业务库单表 > 500 万 / 单库 > 200G 时引导切换
- 分片键由 CommonDB.ds_route 配置驱动，业务代码无感知（ShardingSphere 自动改写 SQL）

---

## 6. 事务边界（多数据源 + 分布式）

### 6.1 同库事务
- 用 Spring `@Transactional`，事务管理器和数据源 1:1
- `RoutingDataSource` 路由结果一旦在事务内确定，整事务锁定该库

### 6.2 跨库事务
**禁止**普通 `@Transactional` 跨数据源——会静默失败。三种合法方案：

| 方案 | 适用 | 缺点 |
|------|------|------|
| **本地消息表** | 大多数业务最终一致 | 实现稍复杂 |
| **Saga（编排式）** | 长流程跨服务 | 调试成本高 |
| **Seata AT 模式** | 强一致诉求 | 性能损耗、运维额外组件 |

> **默认推荐**：本地消息表（详见 07）。Saga / Seata 作为可选 starter。

## 7. ORM 选型

| 候选 | 优点 | 缺点 | 评估 |
|------|------|------|------|
| **MyBatis-Plus 3.5+** | 国内生态成熟、灵活 SQL、与 ShardingSphere 配合好 | 易写出复杂 SQL | **推荐主选** |
| Spring Data JPA | 标准、自动 Repository | 复杂查询难写、与分库分表配合差 | 备选（简单 CRUD 场景） |
| jOOQ | 类型安全 SQL、无魔法 | 需要代码生成、学习成本 | 备选（金融 / 报表） |

**结论**：starter-datasource 默认集成 MyBatis-Plus，预留 JPA / jOOQ 适配。

## 8. 向量库 / RAG 库

### 8.1 选型对比

| 候选 | 优点 | 缺点 | 评估 |
|------|------|------|------|
| **PostgreSQL + pgvector** | 与现有 PG 共集群、运维统一、SQL 友好 | 千万级以上性能下滑 | **推荐起步** |
| Milvus | 专业向量库、亿级性能 | 独立组件、运维成本 | 上量后切 |
| Qdrant | 轻、Rust 实现、过滤能力强 | 国内案例较少 | 备选 |
| Elasticsearch + dense_vector | 检索 + 向量统一 | 内存占用大、向量性能弱于专业库 | 仅在已有 ES 时考虑 |

### 8.2 与业务库的关系

- 向量库**不是业务库**，存的是文档 chunk + embedding + 元数据指针
- 业务库存原文（或对象存储指针），向量库只做相似度召回
- RAG 索引重建是离线作业，不影响在线业务库

### 8.3 多租户隔离

| 隔离级别 | 方式 | 适用 |
|----------|------|------|
| 弱 | 同 collection，metadata 加 tenant_id 过滤 | 中小客户 |
| 中 | 每租户一个 collection | 中大客户 |
| 强 | 每租户独立 Milvus 实例 | 关键客户 / 合规 |

## 9. 数据迁移与版本管理

- **CommonDB schema** 由脚手架统一管理，Flyway 文件放 `starter-datasource/src/main/resources/db/migration/common/`
- **BizDB schema** 由各业务自管，Flyway 路径 `db/migration/biz/`，但脚手架强制扫描规则（命名 `V{yyyyMMdd_HHmm}__{desc}.sql`）
- **回滚策略**：永远向前修复（不写 down 脚本），紧急回滚靠备份恢复
- **大表变更**：强制走 `pt-online-schema-change` / `gh-ost`，禁止裸 `ALTER TABLE`

---

## 10. 缓存层（与多数据源协同）

- starter-cache 提供二级缓存抽象：L1 本地（Caffeine） + L2 远程（Redis）
- 缓存 key 规范：`{app}:{tenant}:{biz}:{entity}:{id}`
- 失效策略：写操作发布"缓存失效事件"到 Redis Pub/Sub，所有实例本地 L1 同步失效
- 默认禁止"无 TTL"缓存，强制最长 TTL（如 24h）

## 12. 待澄清问题

1. "10 个业务库"是物理上的 10 个 PG/MySQL 实例，还是同实例多 schema？影响 ds_pool 设计。
2. 业务库的拆分维度：按**租户**、按**业务域**、还是**租户×业务域**？决定路由表结构。
3. 是否需要"影子库"（生产数据脱敏后给压测用）？影响 ShardingSphere 配置。
4. RAG 文档来源：上传的 PDF / Word？爬取的网页？业务库实时同步？影响 ingestion pipeline。
5. 是否有数据合规要求（GDPR / 个保法）？影响多租户隔离层级和审计粒度。
