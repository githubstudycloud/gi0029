## 2. 中间件抽象（关键：业务代码不感知具体 MQ）

### 2.1 SPI

| 接口 | 职责 |
|------|------|
| `MessageProducer` | 同步 / 异步发送、批量发送 |
| `TxMessageProducer` | 事务消息（与本地事务原子） |
| `MessageConsumer` | 注册监听器、并发度配置 |
| `MessageRouter` | topic / partition / key 计算 |
| `MessageSerializer` | 序列化（默认 JSON，可换 Avro / Proto） |
| `DeadLetterHandler` | 死信处理 |

### 2.2 默认实现 + 可替换

| 中间件 | 适用 | 启用方式 |
|--------|------|----------|
| **Kafka** | 高吞吐、日志流、事件溯源 | `starter-messaging-kafka` |
| **RocketMQ** | 国内、事务消息原生 | `starter-messaging-rocketmq` |
| **RabbitMQ** | 复杂路由、低延迟、传统应用 | `starter-messaging-rabbit` |
| **Redis Stream** | 轻量场景、本地开发 | `starter-messaging-redis` |
| **本地内存** | 单元测试 | 自动激活（test profile） |

> 业务方在 Maven 里多引一个就多一种能力，业务代码完全不变。

## 3. 消息契约（跨服务异步）

### 3.1 契约即代码
- 跨服务消息的 schema 放共享模块 `event-contract/`
- 用 **Avro / Proto** 定义（不用 JSON 自由格式）
- Schema Registry 管理版本（向前 / 向后兼容）

### 3.2 命名规范
```
{domain}.{aggregate}.{action}     例：order.order.created
{domain}.{aggregate}.{action}.v2  显式版本（破坏性变更）
```

### 3.3 包络字段（强制）
所有消息必带：
- `messageId` (UUID v7)
- `eventTime` (ISO8601)
- `producer` (服务名 + 版本)
- `tenantId`
- `traceId`
- `schemaVersion`

## 4. 事务消息（你强调的"业务+消息原子"）

### 4.1 推荐：本地消息表（默认方案）

```
┌────────────┐      事务A：业务表 + 消息表 一起提交
│ BizApp     │      ────────────────────────────►
└──────┬─────┘
       │ 提交后异步轮询 outbox
       ▼
┌────────────┐
│ Outbox     │  ────► 投递到 MQ ────► Consumer
│ Poller     │       (失败重试)
└────────────┘
```

**优势**：与任何 MQ 解耦、强一致、易调试
**劣势**：有 1-2 秒延迟（取决于 poll 间隔）

### 4.2 备选：RocketMQ 事务消息（原生）
- 优势：实时
- 劣势：绑定 RocketMQ，不便于切换

### 4.3 备选：CDC（Debezium 监听 outbox 表）
- 优势：实时、解耦、扩展性好
- 劣势：架构重，需要额外部件

> **默认本地消息表，单独场景启用 CDC。**

## 5. 消费侧契约

### 5.1 幂等
- 强制：消费者必须实现幂等（基于 `messageId` 去重表 / 业务唯一键）
- 脚手架提供 `@Idempotent` 注解，自动用 Redis SetNX 防重

### 5.2 顺序
- 默认：不保证顺序
- 需顺序：用 partition key（Kafka）/ message group（RocketMQ）
- 强顺序：单 partition + 单消费者（损失吞吐）

### 5.3 重试与死信
- 自动重试：3 次（指数退避：1s、5s、25s）
- 仍失败 → 死信队列（DLQ）
- DLQ 监控告警，人工 / 自动补偿

### 5.4 部分失败（与 09 部分失败模式对齐）
- 批量消费时允许"部分成功 + 部分进 DLQ"
- 显式声明 `@PartialFailureAllowed`，不允许默默吞异常

## 6. 异步任务（@Async / 线程池）

### 6.1 线程池规范
- 禁止用 `@Async` 默认线程池
- 脚手架提供 `Executors` 工厂，按业务命名：`taskExecutor("ai-pool")`
- 配置走 Nacos：核心 / 最大线程数、队列大小、拒绝策略
- 监控：每个池暴露 Micrometer 指标

### 6.2 上下文透传
- TaskDecorator 强制透传：TraceContext、用户、租户、数据源 key
- 否则异步任务里无法路由到正确数据源

## 7. 定时任务

| 候选 | 适用 | 优劣 | 评估 |
|------|------|------|------|
| Spring `@Scheduled` | 单机简单任务 | 多实例需自管理锁 | 不推荐生产 |
| ShedLock | 分布式锁基础上的 @Scheduled | 简单可靠 | 简单场景 |
| **XXL-Job** | 国内主流、调度中心、可视化 | 需独立部署 | **推荐主选** |
| Quartz Cluster | 老牌、JDBC 锁 | 配置繁琐 | 不推荐新项目 |
| K8s CronJob | 短生命周期任务 | 与业务隔离 | 离线脚本 |

**推荐组合**：
- 业务定时任务 → XXL-Job
- 框架内部定时（如 outbox poller）→ ShedLock
- 离线 ETL → K8s CronJob

## 8. AI 长任务（与 05 衔接）

- 走通用任务队列：`ai-task-queue`
- 状态机：`PENDING → RUNNING → SUCCESS/FAILED/CANCELED`
- 进度回写（Redis），SSE / WebSocket 推送给前端
- 超时强制中断 + 释放资源（关闭流式连接、回收 token 配额）

---

## 9. Webhook（对外异步通知）

- 出站 Webhook：用消息驱动 → 出站投递服务统一签名 / 重试 / DLQ
- 入站 Webhook：网关侧验签 → 转为内部事件
- 重试策略：指数退避，最长 24h
- 客户端要求：5s 内返回 2xx 才视为成功
