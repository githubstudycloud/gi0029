# 09 · 编码规范与异常处理

> 解决：**圈复杂度 ≤ 4、单方法 ≤ 50 行、业务层禁 catch Exception 但允许全局兜底、部分流程允许部分失败如何设计、规范如何强制**

---

## 1. 编码硬约束（强制，CI 拒绝合并）

| 维度 | 数值 | 说明 | 工具 |
|------|------|------|------|
| 单方法行数 | ≤ 50（含空行注释） | 超出必须拆 | Checkstyle / PMD |
| 圈复杂度 | ≤ 4 | 嵌套 if / 多分支必须拆 | PMD / Sonar |
| 单类公开方法 | ≤ 7（建议）/ ≤ 12（硬上限） | 大类必须拆职责 | PMD |
| 单文件行数 | ≤ 500 | 超出必须拆模块 | Checkstyle |
| 包深度 | ≤ 6 层 | 防止过度嵌套 | ArchUnit |
| 参数个数 | ≤ 5 | 超出用 DTO 包装 | PMD |
| 嵌套深度 | ≤ 3 层 if/for | 超出必须卫语句 / 提取方法 | PMD |
| 字段个数 | ≤ 15 | 大对象拆值对象 | PMD |

**实施关键**：
- 所有规则配置文件放 `build/checkstyle/`、`build/pmd/`，由脚手架统一维护
- 业务方 pom 引入即生效，**不允许业务方覆盖规则**（仅可在 BOM 升级时讨论调整）
- 例外申请走 PR + Issue 审批，加注释 `// suppress:rule-id reason=xxx`

## 2. 命名规范

| 元素 | 规则 | 反例 |
|------|------|------|
| 包名 | 全小写 + 单词 | `userMgr` ❌ |
| 类名 | 大驼峰，业务后缀（Service / Repository / UseCase） | `UserHelper` ⚠️（除 tools 模块外不用 Helper） |
| 方法名 | 动词开头，业务动作 | `processData` ❌（含义模糊） |
| 布尔字段 | `is/has/can` 前缀 | `flag` ❌ |
| 常量 | UPPER_SNAKE | `Max_Count` ❌ |
| DTO | `XxxRequest / XxxResponse / XxxView` | `XxxBean` ❌ |
| Entity | 与表名对应，单数 | `Users` ❌ |

## 3. 异常体系（核心）

### 3.1 异常分类（强制三类，禁止其他根类）

```
            Throwable
                │
              Exception
                │
       ┌────────┼────────────────────────────┐
       │        │                            │
RuntimeException                     CheckedException (基本不用)
       │
   ┌───┴────────────────┬────────────────────┐
   │                    │                    │
BusinessException   SystemException     ThirdPartyException
（业务规则违反）      （系统/基础设施错）   （三方调用失败）
   │                    │                    │
 ProductOutOfStock   DatabaseException   PaymentApiTimeout
 OrderInvalidState   RedisException      WeChatLoginFailed
```

### 3.2 三类异常的职责

| 类型 | 用途 | 是否记 ERROR 日志 | HTTP 状态 | 是否对外暴露 message |
|------|------|-------------------|-----------|----------------------|
| `BusinessException` | 业务规则不满足（库存不足、状态不允许） | 否（INFO/WARN） | 4xx | 是（带 i18n 错误码） |
| `SystemException` | 系统内部错误（DB 连不上、序列化失败） | 是 | 5xx | 否（仅返回 trace_id） |
| `ThirdPartyException` | 三方调用失败（含子类标注是哪个三方） | 是 | 502 / 503 | 否（仅 trace_id） |

### 3.3 错误码规范
```
{业务域}_{模块}_{序号}
USER_AUTH_001       账号不存在
ORDER_CREATE_001    库存不足
SYS_DB_001          数据库连接失败
THIRD_WECHAT_001    微信登录失败
```
- 错误码注册表（YAML / DB）由 starter-kernel 加载
- 同一错误码全公司唯一
- 国际化文案绑定错误码

## 4. "禁止业务层 catch Exception"——精确规则

### 4.1 禁止（ArchUnit 强制）

```java
// 业务层（service / usecase / repository / web）
try {
    doSomething();
} catch (Exception e) {  // ❌ 拒绝
    log.error("xxx", e);
    return null;
}
```

### 4.2 允许的局部捕获（必须满足三个条件之一）

1. **明确捕获具体异常类型**：
```java
try {
    parseJson(s);
} catch (JsonProcessingException e) {  // ✅ 具体类型
    throw new BusinessException("BIZ_PARAM_001", "参数格式错误", e);
}
```

2. **资源清理（try-with-resources 自动）**：
```java
try (var conn = ds.getConnection()) {  // ✅ 优先 try-with-resources
    ...
}
```

3. **三方调用边界（必须包装为 ThirdPartyException）**：
```java
try {
    return thirdClient.call();
} catch (IOException | TimeoutException e) {  // ✅ 具体类型
    throw new ThirdPartyException("THIRD_X_001", "X 服务调用失败", e);
}
```

### 4.3 全局兜底（仅以下两处允许）

| 位置 | 类 | 兜底原因 |
|------|----|----------|
| Web 层 | `@RestControllerAdvice GlobalExceptionHandler` | 防止异常裸奔到客户端 |
| 异步线程顶层 | `Thread.UncaughtExceptionHandler` / `@Async` 包装器 | 防止线程死掉无声 |
| 消息消费者顶层 | `MessageConsumer` 抽象基类 | 决定 ack / nack / DLQ |
| 定时任务顶层 | XXL-Job 执行器封装 | 防止任务进入异常状态 |

兜底处必须：
- 记 ERROR 日志（含 stack）
- 上报 metrics（按异常类型分桶）
- 返回标准错误响应（含 trace_id）

## 5. "部分失败"模式（你特别提到，企业级关键）

### 5.1 场景例子
- 批量导入 1000 条订单，某条业务校验失败 → 跳过该条，继续，最终汇总
- 调用 5 个下游服务做数据聚合，某个服务超时 → 返回部分数据 + 标记降级
- AI 批量摘要 100 篇文档，某篇模型超时 → 部分成功

### 5.2 设计：显式 PartialResult 模型（脚手架内置）

```java
// 伪代码，仅说明形态
public final class PartialResult<T> {
    List<T> success;           // 成功项
    List<FailedItem> failed;   // 失败项（含原因码、错误信息、原始输入引用）
    PartialPolicy policy;      // ALL_OR_NONE / BEST_EFFORT / FAIL_FAST_THRESHOLD
    boolean truncated;         // 是否因阈值提前终止

    public boolean hasFailures() { ... }
    public boolean isCompleteSuccess() { ... }
}
```

### 5.3 三种 PartialPolicy

| 策略 | 行为 | 适用 |
|------|------|------|
| `ALL_OR_NONE` | 任一失败 → 全回滚 | 强一致（如转账批次） |
| `BEST_EFFORT` | 失败收集 + 全部跑完 | 数据导入、AI 批处理 |
| `FAIL_FAST_THRESHOLD` | 失败率 > X% 提前停 | 长任务防雪崩 |

### 5.4 强制规则
- 允许部分失败的方法**必须返回 `PartialResult<T>`**，禁止返回 `List<T>` + 静默吞错
- 调用方**必须显式处理 `failed` 列表**（ArchUnit 检查赋值返回值未读）
- 部分失败必须落审计日志 + 暴露指标

## 6. 业务层与异常的具体配合

### 6.1 业务校验失败 → 主动抛 BusinessException
```java
if (stock < qty) {
    throw new BusinessException("ORDER_CREATE_001", "库存不足", Map.of("stock", stock));
}
```

### 6.2 调用基础设施失败 → 让框架抛 SystemException
- 不要捕获 SQLException 然后再处理；让 Spring 的事务管理 / Repository 抽象往上抛
- 全局 advice 统一翻译为 SystemException

### 6.3 调用三方失败 → 在防腐层（Adapter）翻译为 ThirdPartyException
- starter-* 的客户端封装层负责翻译
- 业务代码看到的是统一的 `ThirdPartyException` 子类

## 7. 日志规范

| 级别 | 用途 | 例 |
|------|------|----|
| ERROR | 系统异常 / 三方异常 | DB 连接失败 |
| WARN | 业务异常（部分） / 降级生效 | 库存不足、走兜底缓存 |
| INFO | 业务关键节点 | 订单创建、登录成功 |
| DEBUG | 调试细节 | SQL 参数 |
| TRACE | 极细 | 单元测试用 |

**强制结构化（JSON 日志）**：
- 必带：`timestamp, level, traceId, spanId, userId?, tenantId?, app, env`
- 业务日志带 `event` 字段，便于 ES 聚合
- 敏感字段必须 `@SensitiveLog` 自动脱敏

## 8. 注释与文档

- public API 必须 javadoc，包含 @since、@throws
- 业务复杂逻辑要注释 **"为什么"**，不要注释"做什么"
- TODO / FIXME 必须带 issue 链接 + owner，否则 CI 拒绝

---

## 9. 单元测试规范

- 命名：`should_{result}_when_{condition}`
- 单测覆盖率：核心业务 ≥ 80%，普通业务 ≥ 60%
- 必须用 AssertJ（链式断言），禁止 JUnit 原生 `assertEquals`
- Mock 用 Mockito，禁止 PowerMock（除遗留外）
- 集成测试用 Testcontainers，禁止依赖外网
- 测试数据用 Builder 模式，禁止散落 `new XXX(...)` 大段构造
