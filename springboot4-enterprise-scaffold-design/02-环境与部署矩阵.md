# 02 · 环境与部署矩阵

> 解决：**四套环境（local / test / beta / pro）分别跑在哪、配置怎么切、镜像与制品如何流转、灰度怎么落**

---

## 1. 环境矩阵总览

| 环境 | Profile | 运行形态 | 编排方式 | 数据 | 流量来源 | 灰度能力 | 谁可发布 |
|------|---------|----------|----------|------|----------|----------|----------|
| **local** | `local` | 单机 Docker / Docker Compose | `docker compose up` | 内置 SQLite/H2 + 本机 PG/Redis 容器 | 开发者本机 | 无 | 开发者自己 |
| **test** | `test` | Docker Swarm 集群 | `docker stack deploy` | 共享 test PG / Redis / Kafka 容器 | QA / 自动化测试 | 无（直接全量） | CI 自动 |
| **beta** | `beta` | K8s 集群（独立 namespace） | Helm + Argo Rollouts | 独立 beta DB（结构同生产） | 内部员工 + 灰度白名单用户 | **金丝雀 + 权重灰度** | 发布平台审批 |
| **pro** | `pro` | K8s 集群（多可用区） | Helm + Argo Rollouts | 生产 DB（多副本 + 备份） | 全量真实用户 | 滚动 + 蓝绿（重大变更） | 多人审批 + 变更窗口 |

> **核心理念**：四套环境用**同一镜像**，只换配置（环境变量 + 配置中心 namespace）。

---

## 2. 镜像与制品流转

```
开发者本机                   CI                   制品库              环境
  ↓                           ↓                    ↓                  ↓
 mvn verify  ──►  build (one image) ──►  Harbor   ──►   local (本地构建跳过)
                                          │           ──►   test  (Swarm pull)
                                          │           ──►   beta  (K8s pull)
                                          └───────────►   pro   (K8s pull, 同 SHA)
```

**关键不变量**：
- 镜像 tag = `git short sha`（不可变）
- 同一 SHA 的镜像在 test/beta/pro 必须是**同一个 layer hash**
- 配置差异不进镜像，只进配置中心 + K8s ConfigMap/Secret

## 3. 各环境技术细节（大方向版）

### 3.1 local（单机 Docker）

```
docker-compose.local.yml
├── app                  # 当前业务服务（mount 源码可热部署）
├── nacos-standalone     # 配置中心
├── postgres-common      # 公共库
├── postgres-biz         # 业务库（默认 1 个，可加）
├── postgres-vector      # pgvector 库
├── redis
├── kafka-kraft          # 单节点 KRaft 模式
└── prometheus + grafana # 可观测性（可选）
```

**设计点**：
- 提供 `make local-up / local-down / local-reset` 三个口子
- AI 模型默认走 OpenAI 兼容代理（如 Ollama 本地），不强制联外网
- 默认数据集种子（seed）与 test 环境保持一致，方便复现 bug

### 3.2 test（Docker Swarm）

**为什么是 Swarm 而不是 K8s**：你明确指出 test 用 Swarm。Swarm 的优势是配置简单、节点要求低，劣势是生态弱、无原生灰度。所以 test 环境**只做全量更新**，不做灰度演练，灰度演练放到 beta。

**部署单元**：`docker stack deploy -c stack.test.yml app-stack`

**关键约束**：
- Swarm 用 overlay 网络，服务发现走 DNS（`tasks.app-name`）
- 配置中心连接信息走 Swarm secret，不进镜像
- CI 自动部署：MR 合并到 `develop` → 构建镜像 → SSH 到 Swarm manager 触发更新

### 3.3 beta（K8s 灰度生产）

**定位**：beta 是"接受真实流量但仅给灰度用户"的预生产，**与 pro 共用真实数据库（或灰度副本）**——这点必须由你拍板。

**两种数据策略对比**：
| 策略 | 优点 | 缺点 | 适用 |
|------|------|------|------|
| beta 共用 pro DB（带租户标） | 真实数据、易复现 | 灰度服务出 bug 可能影响生产数据 | 大多数 SaaS |
| beta 用 pro DB 的实时副本 | 安全 | 数据有延迟、不能写 | 对数据强一致敏感 |

**灰度策略（详见 03 文档）**：
- Header 灰度：`X-Canary-User: true` 路由到 beta
- 权重灰度：5% → 25% → 50% → 100%（Argo Rollouts 自动）
- 用户 ID 哈希灰度：稳定地把同一用户固定在同一版本

### 3.4 pro（K8s 正式）

**最低要求**：
- 多可用区（≥ 2 AZ）
- HPA + PDB + PodAntiAffinity
- 滚动发布默认 `maxSurge=25% maxUnavailable=0`（你强调的"不停服"）
- 关键服务支持蓝绿（订单、支付、鉴权类）
- 所有变更走变更窗口 + 审批

## 4. 配置分层（与环境矩阵对应）

```
优先级（高 → 低）
1. K8s 容器环境变量          ← 仅放敏感开关 / 紧急覆盖
2. Nacos 配置中心 (动态)     ← 业务配置、降级开关、灰度比例
3. application-{env}.yml    ← 仅放该环境固定的、与代码同生命周期的配置
4. application.yml          ← 默认值
```

**Profile 对应**：
| Profile | Nacos namespace | 自动激活方式 |
|---------|-----------------|--------------|
| local | `local` | `SPRING_PROFILES_ACTIVE=local`（开发者 IDE） |
| test | `test` | Swarm secret 注入环境变量 |
| beta | `beta` | K8s Deployment env |
| pro | `pro` | K8s Deployment env + 准入 webhook 校验 |

## 5. 部署制品形态

| 制品 | 内容 | 谁产出 | 谁消费 |
|------|------|--------|--------|
| 业务镜像 | `app:{git-sha}` | 业务 CI | 各环境 |
| Helm Chart | 通用模板（Resource 限额、Probe、HPA、Service、Ingress） | 脚手架团队 | 业务 CD |
| Compose 模板 | local / test 用 | 脚手架团队 | 业务方本机 + CI |
| Argo Rollouts CR | 灰度策略定义（CanarySteps、AnalysisTemplate） | 脚手架团队 | beta / pro |

> **复用关键**：业务方只维护"差异 values.yaml"，公共部分由脚手架统一升级。

---

## 6. 环境间数据流（详见 04）

- **公共库（CommonDB）**：每环境独立一套，但 schema 完全一致；schema 变更走 Flyway，由脚手架统一管理
- **业务库（BizDB）**：每环境独立；schema 由各 BizApp 自管，但目录结构 / 命名规范统一
- **向量库**：建议与对应环境的 BizDB 同集群（pgvector）或独立（Milvus）

## 8. 待澄清问题

1. 是否有"私有化部署给客户"的场景？若有，需要额外一套"customer" profile（单机 Docker 全套），影响打包脚本。
2. test 环境的 Swarm 集群规模？节点数 / 是否多机？影响 stack 文件编写。
3. K8s 是自建还是云托管（EKS/ACK/TKE）？影响 Helm Chart 中 storageClass / ingressClass 的默认值。
4. 是否需要支持"按地域多活"？若有，pro 需要拆成 pro-cn / pro-us 等。
