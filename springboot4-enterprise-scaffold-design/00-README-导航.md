## 2. 核心设计目标（一句话版）

构建一个**可被多个业务团队复用、可灰度演进、可快速淘汰过时依赖、可在 4 套环境一致运行**的 Spring Boot 4.x 企业级脚手架，AI / 多数据源 / 配置总线 / 灰度发布作为一等公民内置。

---

## 3. 设计原则（贯穿所有文档）

1. **接口先行**：跨子项目走 SPI / OpenAPI / 消息契约，绝不走类直连。
2. **可替换优于内置**：所有外部依赖（消息、缓存、数据源、模型、IdP）都经过本地适配层。
3. **配置驱动一切**：环境差异、灰度比例、降级开关、数据源路由全部走配置中心。
4. **失败可见且可控**：业务层禁止 `catch (Exception e)`，但允许"部分失败"显式声明。
5. **依赖收口**：所有依赖必须出现在 `bom` 模块，禁止子模块直接写版本号。
6. **小步快跑**：单方法 ≤ 50 行、圈复杂度 ≤ 4、单类聚焦单一职责。

## 4. 术语表（统一口径，避免后续歧义）

| 术语 | 定义 |
|------|------|
| **脚手架（Scaffold）** | 本仓库整体，提供给业务团队 fork 或依赖的工程骨架 |
| **业务项目（BizApp）** | 基于脚手架派生出的具体业务服务 |
| **共享内核（Shared Kernel）** | 脚手架内不可被业务覆盖的核心模块（异常、日志、上下文等） |
| **能力模块（Starter）** | 脚手架提供的可选能力（AI、多数据源、消息等），通过 `spring.factories` 自动装配 |
| **SPI 边界** | 子项目之间的契约边界，只暴露接口 + DTO，不暴露实现 |
| **公共库 (CommonDB)** | 存放租户、数据源路由、配置元数据的库；所有 BizApp 启动时必读 |
| **业务库 (BizDB[N])** | 真实业务数据所在库，按租户 / 业务域水平拆分 |
| **环境（Env）** | local / test / beta / pro 四级，分别对应 Docker / Swarm / K8s 灰度 / K8s 生产 |
| **接口版本路由** | 网关侧根据 header / path 把同一对外地址路由到不同后端实现 |
| **超期包 (Stale Dep)** | 最近一次发布时间超过 4 年的第三方依赖 |
| **本地化包 (LocalLib)** | 把超期包源码 / Jar 收入仓库 `local-libs/` 目录，通过 `system` scope 或私服镜像引入，逐步替换 |

## 5. 关键全局约束（编码 / 规范类，所有子模块强制）

| 约束 | 数值 / 规则 | 检查手段 |
|------|-------------|----------|
| 单方法行数 | ≤ 50（含空行注释） | Checkstyle / ArchUnit |
| 圈复杂度 | ≤ 4 | PMD / SonarQube |
| 业务层 catch Exception | **禁止** | ArchUnit 自定义规则 |
| 全局兜底 | 仅允许在 `@RestControllerAdvice` 与异步线程顶层 | ArchUnit |
| 单类公开方法 | ≤ 7（建议） | PMD |
| 包深度 | ≤ 6 层 | ArchUnit |
| 依赖发布时间 | ≥ 4 年内 | 自研 maven-enforcer 插件（详见 08） |
| 直接写依赖版本号 | **禁止**（必须由 bom 管） | maven-enforcer |

## 6. 待你确认的"大方向"问题（决定后再展开细节）

> 这里是我希望你先拍板的方向性问题；任意一项变更都会显著影响后续 01–10 的细节。

1. **JDK 基线**：Spring Boot 4.x 需要 JDK 21 起。是否锁定 JDK 21 LTS？还是允许业务方升级到 JDK 25？
2. **构建工具**：Maven（推荐，企业接受度高）还是 Gradle（更灵活但学习曲线陡）？
3. **多项目组织形态**：
   - A. **单仓多模块（Mono-repo）**：脚手架 + 公共能力 + 示例 BizApp 一个仓库
   - B. **脚手架仓 + 业务仓分离**：脚手架发布 BOM + Starter 到私服，业务仓只依赖
   - C. **混合**：脚手架 mono-repo，业务方独立仓但依赖 BOM
4. **配置中心选型**：Nacos（国内主流）/ Apollo（携程系，灰度强）/ Consul + Spring Cloud Config（云原生）/ 自研基于 Git。
5. **网关选型**：Spring Cloud Gateway（Reactive，与 SB4 契合）/ Higress（阿里 Envoy 系）/ APISIX。
6. **AI 框架**：Spring AI 1.x（官方，绑定 SB4）/ LangChain4j（社区活跃，模型多）/ 双适配（成本高）。
7. **K8s 灰度方案**：Istio（功能全但重）/ Argo Rollouts（轻量、专注发布）/ OpenKruise（阿里、滚动能力强）。
8. **超期包策略激进度**：
   - 严格：4 年硬卡，超期立即 local-lib
   - 渐进：4 年警告 + 5 年硬卡，给业务过渡期

> 我的**初步推荐组合**：JDK 21 + Maven + 方案 C（脚手架 mono-repo + 业务独立仓） + Nacos + Spring Cloud Gateway + Spring AI + Argo Rollouts + 渐进策略。
> 你确认或调整后，我再逐篇深化。
