# 06 · 认证鉴权与三方接入

> 解决：**统一身份模型、网关侧鉴权、多种登录方式（账密 / 短信 / SSO）、对接三方 IdP（OIDC / OAuth2 / SAML / 企业微信 / 钉钉 / 飞书）、跨服务身份传递**

## 2. 统一身份模型（关键：先定模型，再做集成）

### 2.1 核心实体

| 实体 | 含义 |
|------|------|
| `Identity` | 一个登录主体（人 / 服务账号 / 机器人） |
| `IdentitySource` | 身份来源：internal / wechat / google / ldap / saml-{provider} |
| `IdentityBinding` | 同一个 Identity 在不同 source 下的绑定关系（一人多登录方式） |
| `Tenant` | 租户 |
| `Role` | 角色（角色绑租户） |
| `Permission` | 权限（操作 + 资源类型） |
| `ResourceScope` | 资源范围（数据级 ACL：行级 / 字段级） |

### 2.2 身份联邦原则
- 不同登录方式登录同一个人 → 同一个 `Identity.id`
- 通过手机号 / 邮箱 / OpenID 关联
- 首次第三方登录时显式询问"绑定到现有账号 / 新建账号"

## 3. 鉴权位置（关键：分层职责）

| 层 | 责任 | 不做的事 |
|----|------|----------|
| **网关** | Token 校验、黑名单、IP / 频次限流、解析 claims 写入 header | 业务规则、数据级权限 |
| **starter-auth (BizApp)** | 资源级鉴权、注解校验、ABAC 引擎 | Token 校验（信任网关） |
| **Service / Repo** | 数据级 ACL（tenant_id 强制注入、行级过滤） | 任何来自请求的鉴权决策 |

> **原则**：网关已校验 Token，BizApp 不重复校验签名（节省性能），但**必须校验 claims 的有效性**（如租户、过期）。BizApp 间内部调用走 mTLS + 服务身份。

## 4. Token 设计

| 类型 | 用途 | 寿命 | 存储 |
|------|------|------|------|
| **Access Token** | 业务接口调用 | 30 min | 客户端持有（前端 Cookie / Header） |
| **Refresh Token** | 刷新 access | 30 天，可滑动 | 客户端 + 服务端 hash 比对 |
| **Service Token** | 服务间调用 | 1 h | 服务启动获取，临近过期自动续签 |
| **One-Time Token** | 三方回调 / 手机验证 | 5 min | Redis |

**格式选型**：
| 候选 | 优点 | 缺点 | 评估 |
|------|------|------|------|
| **JWT (RS256)** | 标准、生态全、无状态 | 撤销难（依赖黑名单） | **推荐** |
| PASETO | 默认安全、无算法选择漏洞 | 生态小、客户端 SDK 少 | 备选 |
| Opaque Token + Redis | 易撤销、可见性强 | 每次需查 Redis | 高安全场景 |

> **混合策略**：Access 用 JWT（高性能），Refresh 用 Opaque（强可控）。

## 5. 三方 IdP 接入（按协议归类）

### 5.1 OIDC 系（推荐主路径）
- 微信开放平台、Google、Apple、Microsoft、Okta
- starter-auth 提供统一 `OidcProvider` 接口
- 配置驱动（Nacos）：

```yaml
auth:
  oidc-providers:
    - name: google
      client-id: xxx
      client-secret-ref: secret://auth/google
      issuer: https://accounts.google.com
      scopes: [openid, email, profile]
      tenant-strategy: domain     # 按邮箱域名分租户
    - name: wechat-work
      client-id: xxx
      ...
```

### 5.2 OAuth2 系（非标准 OIDC）
- 部分国内三方（早期版本微信 / QQ 等）
- 提供专门 adapter，对外仍统一为 `IdentityProvider` SPI

### 5.3 SAML 2.0
- 大型企业 SSO（Okta / OneLogin / 自建 ADFS）
- starter-auth 内置 Spring Security SAML 适配
- 配置元数据 URL 即可

### 5.4 企业 IdP
- 企业微信 / 钉钉 / 飞书：用各自 SDK 封装为 OIDC-like
- LDAP / AD：内置 `LdapBindAuthenticator`
- 自定义 IdP（客户内网）：实现 `IdentityProvider` SPI 即可

### 5.5 设备 / 服务身份（M2M）
- 服务间：mTLS + SPIFFE/SPIRE（K8s 环境优选）
- 设备：客户端证书 + 设备指纹
- API Key（仅给开放平台 / 第三方集成商）

## 6. 鉴权模型（Permission Model）

### 6.1 推荐：RBAC + ABAC 混合

| 层级 | 用途 | 例子 |
|------|------|------|
| **RBAC** | 角色 → 权限粗粒度 | "运营经理"可"导出订单" |
| **ABAC** | 属性条件细粒度 | 仅可导出"自己部门"的订单 |
| **ReBAC（关系）** | 资源所有权 / 协作 | 文档协作者列表 |

### 6.2 实现方式

| 方案 | 优点 | 缺点 | 评估 |
|------|------|------|------|
| Spring Security 表达式 | 内置、零成本 | 复杂条件难维护 | 简单场景 |
| **Casbin** | 配置化、模型灵活 | Spring Boot 4 兼容需要验证 | **推荐** |
| OpenFGA | 关系图鉴权、Google Zanzibar 模型 | 重，需独立服务 | 大型 SaaS |
| 自研规则引擎 | 完全可控 | 维护成本 | 不推荐 |

## 7. 数据级 ACL（关键：必须强制）

- **租户隔离**：MyBatis 拦截器自动注入 `WHERE tenant_id = ?`，业务无需写
- **数据所有权**：Entity 上 `@OwnedBy` 注解，自动校验当前用户是否有权限
- **字段级**：敏感字段 `@Sensitive(role = "admin")`，序列化时按角色脱敏
- **逃生舱**：超管 / 系统任务可显式 `SkipAcl.run(() -> ...)`，但必须审计

## 8. 鉴权失败的处理（与 09 异常处理对齐）

| 情况 | HTTP | 错误码 | 业务处理 |
|------|------|--------|----------|
| Token 缺失 / 无效 | 401 | AUTH_001 | 客户端引导登录 |
| Token 过期 | 401 | AUTH_002 | 客户端用 Refresh 刷新 |
| 权限不足 | 403 | AUTH_101 | 客户端提示无权 |
| 资源不属于该租户 | 404（**不暴露存在性**） | RES_404 | 同 not found |
| 三方 IdP 回调失败 | 302 → 错误页 | OIDC_xxx | 错误页带 trace_id |

## 9. 跨服务身份传递

```
客户端 ──(JWT)──► 网关 ──(解析 claims, 加 header)──► BizApp-A ──(Service Token + 原 user header)──► BizApp-B
```

透传的 header（脚手架统一定义）：
- `X-User-Id`、`X-Tenant-Id`、`X-User-Roles`（短列表，长则查缓存）
- `X-Request-Id`（trace 用，详见 10）
- `X-Original-Token`（仅在需要 IdP 二次校验时透传，默认不传）

## 11. 待澄清问题

1. 用户基数？决定 Token 缓存策略和 Auth Service 横向扩展。
2. 是否有"代理登录"（管理员模拟用户）需求？影响审计与 Token 设计。
3. 是否需要"单点登出"（一次登出所有第三方）？需要全局 session 中心。
4. 是否要支持 Passkey / WebAuthn？2026 年企业级开始普及，建议预留。
5. 是否对接政务 / 金融行业的 CA 体系（如国密 SM2）？影响加密算法栈。
