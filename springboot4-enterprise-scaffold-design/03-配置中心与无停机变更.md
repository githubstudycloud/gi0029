## 2. 配置三类划分（重要：决定推送策略）

| 类别 | 示例 | 来源 | 变更方式 | 是否需要重启 |
|------|------|------|----------|--------------|
| **静态配置** | JDBC URL、Nacos 地址、应用名 | application-{env}.yml + K8s env | 改完发版 | 是（但走滚动） |
| **动态配置** | 业务开关、灰度比例、限流阈值、AI 模型选择 | Nacos | 推送即生效（`@RefreshScope`） | **否** |
| **路由配置** | 网关路径 → 后端服务映射、接口版本路由表 | Nacos `gateway-routes` | 推送即生效，网关动态加载 | **否** |

## 3. 你提到的"配置到 Gitlab 或表"——双源方案

### 3.1 双源的角色分工

| 来源 | 用途 | 优势 | 谁来改 |
|------|------|------|--------|
| **Gitlab Repo**（yaml 文件） | 长期配置、审计要求高、需要 PR 评审 | 全部走 Git 流程，可追溯、可回滚到任意 commit | 架构 / 运维 |
| **DB 配置表** | 临时调参、运营配置、灰度比例 | 改了立刻生效，运营平台直接写 | 业务运营 / 平台 |

### 3.2 同步链路

```
Gitlab repo  ──webhook──►  config-sync-job  ──API──►  Nacos
                                  ▲
DB 配置表  ──CDC/定时扫描──────────┘

Nacos  ──long polling──►  各 BizApp / Gateway
```

**关键约束**：
- 同一配置 Key 不能同时在两个源出现，启动时校验，冲突即报警
- DB 表写入时同步推一次到 Nacos（旁路双写），避免 CDC 延迟
- Gitlab 是"主"——上线必走 Git，DB 是"运营态调参"，重大变更必须沉淀回 Git

## 4. 网关如何"保留对外地址，更换后台接口"

> 你描述的是典型的**接口版本路由**——对外 URL 不变，后端可平滑切换。

### 4.1 路由规则数据结构（存 Nacos）

```yaml
# 示例：网关路由规则
- id: order-create
  predicates:
    - Path=/api/order/create
    - Header=X-Api-Version,v2  # 可选
  filters:
    - StripPrefix=1
    - VersionRoute:
        rules:
          - match: { header: X-Tenant, value: tenant-a }
            target: order-service-v2
          - match: { weight: 10 }              # 10% 流量去新版
            target: order-service-v3-canary
          - default: order-service-v2          # 兜底
```

### 4.2 切换流程（无需重启网关）

```
1. 新后端实例上线 (order-service-v3-canary)，注册到注册中心
2. 在 Nacos 改路由规则：weight=10 → 25 → 50 → 100
3. 网关 watch Nacos 变更，热加载路由表
4. 旧版 (v2) 流量降到 0，下线实例
5. 全过程对外 URL `/api/order/create` 不变
```

### 4.3 与 Nginx 对比（你问到的"Nginx 也能但要重启"）

| 方案 | 路由变更生效 | 灰度颗粒度 | 谁能改 | 我的评估 |
|------|--------------|------------|--------|----------|
| Nginx reload | 秒级，但 reload 有副作用 | 上游权重粗粒度 | 运维 | 不推荐做动态路由 |
| Nginx + Lua + Consul | 实时 | 中 | 平台 | 学习曲线陡 |
| Spring Cloud Gateway + Nacos | 实时（毫秒） | 任意（Header / 用户 / 比例） | 平台 / 运营 | **推荐** |
| Higress / APISIX + Dashboard | 实时 | 任意 | 运营 UI | 替代选项 |

## 5. 滚动重启不停服（你强调的"慢慢重启一部分，关一部分"）

### 5.1 K8s 默认能力（已经满足"不停服"）

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 25%        # 临时多起 25% 实例
    maxUnavailable: 0    # 不允许任何实例不可用
```

加上必备配置：
- `readinessProbe`：实例就绪才接流量
- `preStop` hook：sleep 15s 让 LB 摘除
- `terminationGracePeriodSeconds: 60`：优雅关闭
- `PodDisruptionBudget`：保证最小可用副本数

### 5.2 Argo Rollouts 进阶（你想要的"慢慢"）

```yaml
strategy:
  canary:
    steps:
      - setWeight: 5
      - pause: { duration: 5m }
      - analysis: { templates: [error-rate, latency] }  # 自动分析
      - setWeight: 25
      - pause: { duration: 10m }
      - setWeight: 50
      - pause: { duration: 10m }
      - setWeight: 100
```

**自动回滚**：`AnalysisTemplate` 检测错误率 > 阈值 / 延迟超 SLO，自动回滚到上一版本。

### 5.3 与 Swarm 的差异（test 环境）

Swarm 也有 rolling update：
```yaml
update_config:
  parallelism: 1       # 一次更 1 个
  delay: 10s
  failure_action: rollback
  order: start-first   # 先起新再停旧
```
但**没有就绪探针**（Swarm 仅 healthcheck），所以 test 环境不能完美无停机，可接受（因为 test 不承载真实流量）。

## 6. 配置变更"缓慢生效"的两条路径

| 场景 | 方案 |
|------|------|
| **代码版本更新**（需重新部署） | Argo Rollouts 灰度（如上） |
| **配置变更**（同代码版本，行为切换） | 配置中心灰度发布（Nacos Beta 发布 / 标签灰度） |

> Nacos 支持"灰度发布"：先推给打了 `gray=true` 标签的实例，验证后再全量。这样可以**完全无重启**地小步验证配置变更。

## 7. 配置审计与回滚

- 所有 Nacos 变更必须记录：操作人、时间、diff、原因（强制必填）
- 关键 Key 变更（路由、限流、熔断阈值）触发 Webhook 通知到企业 IM
- 一键回滚到任意历史版本（Nacos 自带 30 天历史，超期归档到对象存储）
- Gitlab 源天然有 commit 历史，DB 源需要单独 audit_log 表
