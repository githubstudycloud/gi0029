# 数据持久化与多租户

## 1. 目标

数据层需要支持多租户、多数据源、读写分离、统一审计字段、统一事务边界和可观测路由。首版重点是稳定和可排障，不把分库分表作为默认能力。

## 2. 数据拓扑

```text
                 ┌────────────────────┐
                 │      CommonDB       │
                 │ tenant/ds/config    │
                 └─────────┬──────────┘
                           │
            ┌──────────────┼──────────────┐
            │              │              │
┌───────────▼────┐ ┌───────▼────────┐ ┌───▼────────────┐
│ BizDB tenant A │ │ BizDB tenant B │ │ BizDB shared   │
└────────────────┘ └────────────────┘ └────────────────┘
```

CommonDB 保存平台元数据，BizDB 保存业务数据。平台不鼓励业务服务直接跨租户、跨库查询。

## 3. CommonDB 最小表

| 表 | 说明 |
| --- | --- |
| `tenant` | 租户基本信息、状态、套餐、隔离等级 |
| `tenant_datasource` | 租户到数据源的映射 |
| `datasource_instance` | 数据源连接信息引用，不保存明文密码 |
| `config_registry` | 配置元数据和分发状态 |
| `idp_binding` | 租户身份源绑定 |
| `audit_event` | 管理操作审计 |

敏感字段只保存 secret 引用，例如 `secretRef`，真实密码由 Secret 管理系统提供。

## 4. 多租户模式

| 模式 | 适用场景 | 首版支持 |
| --- | --- | --- |
| 共享库共享表 | 小租户、低隔离需求 | 支持 |
| 共享库分 schema | 中等隔离 | 可选 |
| 独立库 | 大租户、强隔离 | 支持 |
| 独立集群 | 金融/政企强隔离 | 预留 |

默认推荐：共享库共享表 + 大租户独立库。所有业务表必须包含 `tenant_id`，除非明确声明为平台全局表。

## 5. 数据源路由

路由上下文来源优先级：

1. 显式注解：`@UseDatasource("xxx")`，只允许基础设施层少量使用。
2. 租户上下文：根据 `tenantId` 查询 `tenant_datasource`。
3. 读写上下文：根据事务只读属性或注解选择 read/write。
4. 默认数据源：仅用于启动和公共表。

约束：

1. Controller 不允许手动切换数据源。
2. 数据源切换必须在事务开启前完成。
3. 同一事务内禁止切换到另一个 BizDB。
4. 每次路由结果写入 MDC 和 trace attributes。

## 6. 事务边界

| 场景 | 策略 |
| --- | --- |
| 单库单事务 | 使用 Spring `@Transactional` |
| 同租户多表 | 同库事务 |
| 跨租户写入 | 默认禁止 |
| 跨库写入 | 使用 Outbox、Saga 或补偿 |
| 强一致跨库 | 需要架构评审，不作为默认能力 |

默认不引入 XA 和 Seata。它们能解决一部分一致性问题，但会增加运维、性能和故障恢复成本。

## 7. ORM 选择

原设计默认 MyBatis-Plus + ShardingSphere。优化后建议先对 MyBatis-Plus 与 MyBatis-Flex 做兼容性验证，再确定默认 ORM。

评估维度：

| 维度 | 要求 |
| --- | --- |
| Boot 4 兼容 | 自动配置、mapper 扫描、事务无问题 |
| Jakarta 兼容 | 无旧 `javax.*` 阻塞 |
| 分页 | PostgreSQL/MySQL 均可用 |
| 审计字段 | 创建人、修改人、租户字段自动填充 |
| SQL 可控 | 支持复杂查询但能限制坏味道 |
| 测试支持 | Testcontainers 集成简单 |

## 8. 读写分离

首版建议实现轻量读写路由：

1. 写事务走 primary。
2. 只读事务可走 replica。
3. 强一致读通过注解强制 primary。
4. replica 延迟超过阈值时自动降级到 primary。

ShardingSphere 作为可选 starter，适用于后续分片、影子库、加密列等高级场景。

## 9. 缓存

| 缓存 | 用途 | 约束 |
| --- | --- | --- |
| Caffeine | 单机热点配置、本地短 TTL | 不存强一致业务数据 |
| Redis | 分布式缓存、幂等、锁、限流 | key 必须带应用和租户维度 |

缓存 key 规范：

```text
{app}:{env}:{tenantId}:{domain}:{type}:{id}
```

禁止缓存：

1. 未脱敏个人敏感信息。
2. 无 TTL 的业务对象。
3. 未带租户维度的租户数据。

## 10. 向量库与 RAG 数据

首版使用 PostgreSQL + pgvector，原因是运维简单、事务和权限模型一致，适合从 0 到 1。

升级到 Milvus 或 Elasticsearch Vector 的触发条件：

1. 向量规模达到千万级。
2. 检索延迟无法满足 SLO。
3. 需要专用 ANN 索引能力。
4. RAG 数据与业务库生命周期明显不同。

## 11. 数据迁移

1. 使用 Flyway 或 Liquibase 管理 schema。
2. 平台表和业务表迁移脚本分目录管理。
3. 每个迁移必须可重复执行测试。
4. beta 先迁移，验证后再进入 pro。
5. 破坏性变更必须采用 expand-contract 模式。

## 12. 验收标准

1. 本地可启动 CommonDB + 一个 BizDB。
2. 根据租户自动路由数据源。
3. 事务内切换数据源会被测试拦截。
4. SQL 日志和 trace 能看到 `tenantId` 与 `dsKey`。
5. Redis key 检查规则能阻止无租户 key。
