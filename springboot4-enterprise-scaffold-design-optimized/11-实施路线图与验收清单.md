# 实施路线图与验收清单

## 1. 实施策略

采用分阶段交付。每个阶段都必须有可运行 sample、自动化测试和验收标准。不要在平台能力未稳定前让业务项目大规模接入。

## 2. Phase 0：技术验证

周期：1-2 周。

目标：验证 Spring Boot 4 与关键三方组件兼容性。

交付：

1. 最小 Maven mono-repo。
2. JDK 21 + Spring Boot 4 sample。
3. Spring Cloud Gateway sample。
4. Nacos 配置加载和刷新 sample。
5. MyBatis 系 ORM sample。
6. Kafka 生产消费 sample。
7. Spring AI chat/embedding sample。
8. OpenTelemetry trace sample。

验收：

1. `mvn verify` 通过。
2. 所有 sample 可本地启动。
3. 兼容性风险记录到风险清单。
4. 无法兼容的组件给出替代方案。

## 3. Phase 1：平台核心

周期：2-4 周。

交付：

1. `platform-bom`。
2. `core-context`。
3. `core-error`。
4. `core-web`。
5. `core-logging`。
6. `core-validation`。
7. Checkstyle/PMD/ArchUnit/Enforcer。
8. `sample-basic-app`。

验收：

1. 统一响应和错误码可用。
2. traceId/requestId/tenantId 进入日志。
3. 禁止业务层 `catch Exception` 的 ArchUnit 规则生效。
4. 子模块直接写版本会构建失败。
5. sample-basic-app 通过集成测试。

## 4. Phase 2：基础设施 starter

周期：3-5 周。

交付：

1. `starter-web`。
2. `starter-security`。
3. `starter-config`。
4. `starter-datasource`。
5. `starter-cache`。
6. `starter-observability`。
7. Docker Compose local 环境。
8. Helm 基础模板。

验收：

1. OIDC Resource Server 可校验 token。
2. Nacos 配置可加载和刷新。
3. CommonDB + BizDB 路由可用。
4. Redis 缓存 key 规范检查可用。
5. Prometheus/Grafana/Loki/Tempo 可看到样例数据。

## 5. Phase 3：消息与灰度

周期：3-4 周。

交付：

1. `starter-messaging-kafka`。
2. Outbox relay。
3. 幂等消费组件。
4. 死信查询和重放接口。
5. Argo Rollouts 模板。
6. `sample-messaging`。

验收：

1. 业务事务和 Outbox 写入一致。
2. relay 重启后不丢消息。
3. 重复消费不会产生重复副作用。
4. beta 环境可执行 5% → 25% → 50% → 100% 灰度。
5. 指标异常能自动暂停发布。

## 6. Phase 4：AI 与 RAG

周期：3-6 周。

交付：

1. `starter-ai`。
2. Provider SPI。
3. pgvector 集成。
4. Markdown ingestion。
5. 基础 RAG 问答。
6. Token 成本统计。
7. `sample-ai-rag`。

验收：

1. chat、embedding、RAG 查询可用。
2. RAG 检索按租户隔离。
3. 输出带引用来源。
4. token 用量按租户统计。
5. provider 异常可 fallback 或返回明确错误。

## 7. Phase 5：治理增强

周期：持续迭代。

交付：

1. 业务自助配置平台。
2. 依赖风险看板。
3. local-libs 生命周期看板。
4. SLO 看板。
5. 回滚演练自动化。
6. 可选 RocketMQ、ShardingSphere、SAML、XXL-Job starter。

## 8. 总体验收清单

| 类别 | 验收项 |
| --- | --- |
| 构建 | 新业务项目 10 分钟内从 archetype 创建并启动 |
| 依赖 | 所有依赖版本由 BOM 管理 |
| 质量 | `mvn verify` 包含单测、集成测试、架构测试、依赖检查 |
| 配置 | 配置变更有审批、diff、灰度、回滚 |
| 安全 | OIDC 鉴权、租户上下文、权限注解可用 |
| 数据 | 多租户路由、事务约束、审计字段可用 |
| 消息 | Outbox、幂等、死信、重放可用 |
| 观测 | 日志、指标、trace 自动接入 |
| 部署 | beta/pro 支持 Helm + Argo Rollouts |
| 文档 | 每个 starter 有 README 和 sample |

## 9. 主要风险

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| Boot 4 生态兼容性不足 | 阻塞 starter 落地 | Phase 0 先 spike，必要时替换组件 |
| Nacos 适配滞后 | 配置中心方案受影响 | 保留 Spring Cloud Config 适配 |
| 质量规则过严 | 存量接入困难 | 新项目强制，存量设迁移基线 |
| AI 成本不可控 | 费用风险 | 租户限额、预算、降级 |
| 多数据源排障复杂 | 生产事故风险 | trace 记录 dsKey，禁止事务内切库 |
| local-libs 变长期仓库 | 技术债累积 | owner、到期日、季度清理 |

## 10. 最少待确认问题

只保留会阻塞实施的决策：

1. 是否确认 JDK 21 + Maven 为首版唯一基线。
2. 是否已有 Nacos、Kafka、Redis、PostgreSQL、制品私服基础设施。
3. 是否必须支持 MySQL 作为首版默认数据库。
4. 首批接入业务是否需要 SAML、RocketMQ 或分库分表。
5. AI 能力是否需要首版上线，还是作为 Phase 4 独立交付。
