# 消息异步与任务调度

## 1. 目标

消息模块需要统一事件契约、生产消费、重试、死信、幂等、事务外发和观测。首版默认 Kafka，RocketMQ/RabbitMQ 通过后续 starter 扩展。

## 2. 消息抽象

业务代码面向平台接口：

```java
public interface EventPublisher {
    void publish(DomainEvent event);
}

public interface DomainEvent {
    String eventId();
    String eventType();
    String aggregateId();
    Instant occurredAt();
}
```

不允许业务直接散落调用 KafkaTemplate。具体中间件由 `starter-messaging-kafka` 适配。

## 3. 消息包络

统一字段：

| 字段 | 说明 |
| --- | --- |
| `eventId` | 全局唯一 |
| `eventType` | 事件类型 |
| `eventVersion` | 事件版本 |
| `tenantId` | 租户 |
| `aggregateType` | 聚合类型 |
| `aggregateId` | 聚合 ID |
| `occurredAt` | 业务发生时间 |
| `traceId` | 链路 ID |
| `producer` | 生产服务 |
| `payload` | 业务数据 |

命名规范：

```text
topic: {env}.{domain}.{event-name}
group: {app}.{consumer-purpose}
eventType: {Domain}.{Entity}.{Action}.v{version}
```

## 4. 事务消息策略

默认使用 Outbox：

```text
business transaction
  ├── write business table
  └── write outbox_event
        └── relay job publishes to Kafka
              └── mark published
```

优点：

1. 不绑定具体 MQ 的事务能力。
2. 数据库事务保证业务数据和待发送事件一致。
3. relay 可重试、可观测、可补偿。

Outbox 表关键字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 事件 ID |
| `topic` | 目标 topic |
| `event_type` | 类型 |
| `payload` | 内容 |
| `status` | NEW/PUBLISHING/PUBLISHED/FAILED |
| `retry_count` | 重试次数 |
| `next_retry_at` | 下次重试时间 |
| `created_at` | 创建时间 |
| `published_at` | 发布时间 |

## 5. 消费幂等

消费侧必须具备幂等能力。推荐使用 `processed_event` 表或 Redis 幂等键。

策略：

1. 先登记 eventId，再执行业务。
2. 已处理事件直接 ack。
3. 业务失败按异常类型决定重试或死信。
4. 幂等记录必须带 consumer group。

## 6. 重试与死信

| 异常类型 | 策略 |
| --- | --- |
| 参数非法 | 不重试，直接死信 |
| 下游限流 | 延迟重试 |
| 网络超时 | 指数退避重试 |
| 数据不存在 | 按业务决定延迟或死信 |
| 未知异常 | 有限重试后死信 |

死信处理要求：

1. 保留原始消息。
2. 记录错误堆栈摘要。
3. 支持人工重放。
4. 重放必须审计。

## 7. 顺序消息

默认不承诺全局顺序。需要顺序时：

1. 按 aggregateId 作为 partition key。
2. 单聚合内顺序处理。
3. 消费端避免并发处理同一 aggregateId。
4. 文档明确顺序保证范围。

## 8. 异步线程池

线程池配置：

```yaml
platform:
  async:
    pools:
      default:
        core-size: 8
        max-size: 32
        queue-capacity: 1000
        reject-policy: caller-runs
```

要求：

1. 禁止直接使用 `new Thread`。
2. 禁止使用无界队列。
3. 必须透传 MDC、trace、tenant、security context。
4. 拒绝策略必须可观测。

## 9. 定时任务

首版支持两类：

| 类型 | 适用场景 | 推荐 |
| --- | --- | --- |
| K8s CronJob | 平台维护任务、低频批处理 | 首选 |
| XXL-Job | 业务可视化调度、人工触发、分片任务 | 可选 starter |

不建议新项目默认使用 Quartz Cluster，除非已有成熟运维经验。

## 10. Webhook

对外通知必须具备：

1. 签名。
2. 时间戳防重放。
3. 幂等 eventId。
4. 指数退避重试。
5. 订阅方失败隔离。
6. 投递审计和人工重放。

## 11. 验收标准

1. 业务事务提交后事件可靠进入 Outbox。
2. relay 宕机恢复后能继续发送。
3. 消费重复消息不会重复执行业务副作用。
4. 死信可查询、可重放、可审计。
5. trace 能串联 HTTP 请求、Outbox、Kafka 生产和消费。
