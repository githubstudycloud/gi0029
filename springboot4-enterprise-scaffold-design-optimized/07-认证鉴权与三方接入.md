# 认证鉴权与三方接入

## 1. 目标

认证鉴权模块需要统一用户身份、租户身份、服务身份、权限模型和三方 IdP 接入。外部身份源可以变化，但业务系统内部只面对统一身份模型。

## 2. 身份模型

核心实体：

| 实体 | 说明 |
| --- | --- |
| `Tenant` | 租户 |
| `User` | 平台用户 |
| `Account` | 用户在某租户下的账号 |
| `IdentityProvider` | 身份源，如 OIDC、SAML、企业微信、钉钉 |
| `ExternalIdentity` | 外部身份与内部账号绑定 |
| `Role` | 角色 |
| `Permission` | 权限点 |
| `Policy` | ABAC 策略 |
| `ServiceAccount` | 服务间调用身份 |

原则：

1. 一个 `User` 可加入多个租户。
2. 一个租户可绑定多个 IdP。
3. 外部身份不直接作为业务主键。
4. 所有权限判断最终基于内部账号和租户上下文。

## 3. 认证架构

```text
Client
  └── Gateway
        ├── OIDC login / token validation
        ├── rate limit
        ├── tenant resolution
        └── forward identity headers
              └── BizApp Resource Server
                    └── method/data permission
```

网关负责统一入口认证，业务服务仍必须作为 Resource Server 验证 token 或内部签名，不能只信任裸 header。

## 4. Token 策略

| Token | 用途 | 策略 |
| --- | --- | --- |
| Access Token | API 调用 | JWT，短有效期 |
| Refresh Token | 刷新会话 | 仅认证中心持有，长有效期，可撤销 |
| Service Token | 服务间调用 | client credentials 或 mTLS 绑定 |
| Temporary Token | 文件下载、临时授权 | 极短有效期，一次性 |

JWT claims 最小集合：

```json
{
  "sub": "user-id",
  "tenant_id": "tenant-id",
  "account_id": "account-id",
  "roles": ["admin"],
  "scope": "order:read order:write",
  "sid": "session-id"
}
```

## 5. 三方 IdP 接入

优先级：

1. OIDC：默认主路径，适配 Keycloak、Auth0、Azure AD、Okta 等。
2. OAuth2：适配非标准登录平台。
3. SAML 2.0：政企和存量系统需要时启用。
4. 企业 IdP：企业微信、钉钉、飞书等通过 adapter 接入。
5. LDAP/AD：作为企业目录同步来源，不建议直接作为现代 Web 登录主路径。

IdP 配置按租户隔离：

```yaml
platform:
  security:
    tenants:
      t001:
        idps:
          - code: keycloak
            type: oidc
            issuer-uri: https://idp.example.com/realms/demo
            client-id: demo
            client-secret-ref: secret/keycloak/demo
```

## 6. 授权模型

采用 RBAC + ABAC：

1. RBAC 负责菜单、API、功能点。
2. ABAC 负责数据范围、资源属性、环境条件。
3. 数据权限默认在查询入口和 repository 层双重约束。

权限命名：

```text
{domain}:{resource}:{action}
order:order:read
order:order:create
order:refund:approve
```

## 7. 数据权限

数据权限必须显式建模，不能散落在业务 SQL 字符串中。

策略来源：

1. 租户隔离：`tenant_id = currentTenant`。
2. 组织范围：本人、部门、部门及下级、全部。
3. 资源归属：owner、creator、assignee。
4. 标签或属性：region、level、dataClass。

落地方式：

1. 查询对象带 `DataScope`。
2. Repository 层应用 scope。
3. 复杂报表使用受控 SQL 模板。
4. 所有绕过数据权限的查询必须标注并审计。

## 8. 服务间调用

1. 内部服务使用 client credentials 或 mTLS。
2. 透传用户上下文时必须区分 `actor` 和 `subject`。
3. 禁止服务伪造用户 header。
4. 内部 token 必须有 audience 限制。

## 9. 鉴权失败响应

| 场景 | HTTP | 错误码 |
| --- | --- | --- |
| 未登录 | 401 | `AUTH_UNAUTHENTICATED` |
| token 过期 | 401 | `AUTH_TOKEN_EXPIRED` |
| 权限不足 | 403 | `AUTH_FORBIDDEN` |
| 租户无权访问 | 403 | `AUTH_TENANT_DENIED` |
| 数据不存在或不可见 | 404 或 403 | 按业务安全策略 |

## 10. 首版范围

v1 必做：

1. OIDC Resource Server。
2. JWT 校验和身份上下文。
3. RBAC 权限注解。
4. 租户解析。
5. 数据权限基础模型。

v1 可选：

1. SAML。
2. 企业微信/钉钉/飞书登录。
3. Casbin 或其他策略引擎完整后台。
4. mTLS 自动证书轮换。
