# 环境部署与配置治理

## 1. 环境矩阵

| 环境 | 目标 | 部署方式 | 数据 | 发布策略 | 说明 |
| --- | --- | --- | --- | --- | --- |
| local | 开发自测 | Docker Compose + Maven | 本地容器 | 手动启动 | 最小依赖，便于快速调试 |
| test | 集成测试 | Docker Compose 或轻量 K8s | 共享测试库 | CI 自动部署 | 不引入复杂灰度 |
| beta | 预生产验证 | K8s + Helm + Argo Rollouts | 独立 beta 数据 | 金丝雀 | 接近生产拓扑 |
| pro | 生产 | K8s + Helm + Argo Rollouts | 生产数据 | 滚动/金丝雀/蓝绿 | 多人审批和回滚预案 |

优化点：test 环境不强制 Docker Swarm。Swarm 在当前企业新项目中收益有限，会增加一套运维模型。若已有 Swarm 基础设施可保留，否则建议 test 直接使用 Compose 或轻量 K8s。

## 2. 制品流转

```text
commit
  └── CI: test + quality gate + image build
        └── push image to registry
              └── deploy test
                    └── approve
                          └── deploy beta canary
                                └── automated analysis
                                      └── deploy pro
```

制品要求：

1. 同一个镜像在 test、beta、pro 流转，不重新构建。
2. 环境差异只来自配置和密钥，不来自代码分支。
3. 镜像 tag 必须包含 git commit、构建号和版本号。
4. Helm chart 与应用版本分离，避免应用发版被 chart 变更阻塞。

## 3. 配置分层

| 层级 | 内容 | 主源 | 分发 | 是否可热更新 |
| --- | --- | --- | --- | --- |
| 构建配置 | 依赖版本、插件版本 | Git | Maven | 否 |
| 启动配置 | 应用名、端口、配置中心地址 | Git/Helm values | K8s env/ConfigMap | 否 |
| 运行配置 | 开关、限流、模型选择、路由权重 | Git 审批后同步 | Nacos | 是 |
| 敏感配置 | 密码、token、证书 | Secret 管理系统 | K8s Secret/External Secrets | 部分 |
| 业务自助配置 | 运营开关、租户配置 | DB + 审批流 | Nacos | 是 |

原则：

1. Git 是可审计主源。
2. Nacos 是运行时分发面。
3. DB 承载业务自助平台，不绕过审计直接改线上最终配置。
4. 敏感配置不进入 Nacos 明文。

## 4. Nacos 使用边界

适合放入 Nacos：

1. 业务开关。
2. 灰度权重。
3. 网关动态路由。
4. 限流阈值。
5. AI provider 和模型选择。
6. 线程池参数。

不适合放入 Nacos：

1. 数据库密码。
2. 私钥。
3. 大体积规则文件。
4. 必须强一致的交易参数。
5. 未经审批的临时配置。

## 5. 配置发布流程

```text
change request
  └── diff generated
        └── review/approve
              └── sync to Nacos namespace/group
                    └── gray publish
                          └── observe metrics
                                └── full publish or rollback
```

强制审计字段：

| 字段 | 说明 |
| --- | --- |
| `changeId` | 变更单号 |
| `operator` | 操作人 |
| `approver` | 审批人 |
| `reason` | 变更原因 |
| `diff` | 配置差异 |
| `scope` | 环境、应用、租户、实例标签 |
| `rollbackVersion` | 回滚版本 |

## 6. 网关动态路由

路由规则建议结构：

```yaml
routes:
  - id: order-api-v1
    path: /api/orders/**
    methods: [GET, POST]
    predicates:
      header:
        x-api-version: v1
    targets:
      - service: order-service-v1
        weight: 90
      - service: order-service-v2
        weight: 10
    auth:
      required: true
      scopes: [order:read, order:write]
    rateLimit:
      key: tenant
      qps: 100
```

约束：

1. 路由变更必须经过语法校验。
2. 变更发布前先在 beta namespace 验证。
3. pro 权重变化必须有自动回滚指标。
4. 路由规则必须记录版本，避免热更新后不可追踪。

## 7. 灰度与回滚

Argo Rollouts 推荐步骤：

| 阶段 | 流量 | 验证 |
| --- | --- | --- |
| step 1 | 5% | 错误率、P95、核心业务成功率 |
| step 2 | 25% | 同上 + 日志异常趋势 |
| step 3 | 50% | 同上 + 资源水位 |
| step 4 | 100% | 持续观察 |

自动回滚条件：

1. 5xx 错误率超过基线。
2. P95 延迟超过 SLO。
3. 核心业务成功率下降。
4. JVM OOM、容器频繁重启。
5. 自定义业务指标异常。

## 8. 环境验收

| 项 | local | test | beta | pro |
| --- | --- | --- | --- | --- |
| 一键启动 | 必须 | 可选 | 不适用 | 不适用 |
| 自动化测试 | 必须 | 必须 | 必须 | 必须 |
| 灰度发布 | 不需要 | 不需要 | 必须 | 必须 |
| 配置热更新 | 可选 | 必须 | 必须 | 必须 |
| 观测接入 | 最小 | 必须 | 必须 | 必须 |
| 回滚演练 | 不需要 | 可选 | 必须 | 必须 |
