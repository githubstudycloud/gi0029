# 可观测性与运维基线

## 1. 目标

平台必须默认可观测。业务团队引入 starter 后，HTTP、数据库、缓存、消息、AI、任务调度都应自动暴露日志、指标和链路追踪。

## 2. 三件套

| 类型 | 默认方案 | 说明 |
| --- | --- | --- |
| Logs | JSON logs + Loki | 结构化日志，便于按 traceId/tenantId 查询 |
| Metrics | Micrometer + Prometheus + Grafana | 应用、JVM、业务指标 |
| Traces | OpenTelemetry + Tempo/Jaeger | 跨 HTTP、DB、Redis、Kafka、AI 调用 |

## 3. 日志字段

必备字段：

| 字段 | 说明 |
| --- | --- |
| `timestamp` | 时间 |
| `level` | 日志级别 |
| `app` | 应用名 |
| `env` | 环境 |
| `traceId` | 链路 ID |
| `spanId` | span ID |
| `requestId` | 请求 ID |
| `tenantId` | 租户 |
| `userId` | 用户 |
| `dsKey` | 数据源路由结果 |
| `message` | 日志内容 |
| `errorCode` | 平台错误码 |

## 4. 指标

平台自动暴露：

| 指标 | 维度 |
| --- | --- |
| HTTP 请求数、错误率、延迟 | app、route、method、status |
| DB 连接池 | app、dsKey |
| SQL 耗时 | app、dsKey、operation |
| Redis 命令耗时 | app、command |
| Kafka 生产/消费 | topic、group、status |
| Outbox 堆积 | app、topic、status |
| AI 调用耗时和 token | provider、model、tenant |
| 线程池队列 | pool、app |
| JVM | heap、gc、threads |

标签治理：

1. 禁止把 userId、orderId 等高基数字段作为 metrics label。
2. tenantId 只在必要业务指标中使用，避免 Prometheus 高基数爆炸。
3. 高基数信息进入日志和 trace，不进入通用 metrics。

## 5. 链路追踪

必须串联：

1. 网关入口。
2. BizApp HTTP 处理。
3. 数据库查询。
4. Redis 访问。
5. Kafka 生产和消费。
6. Outbox relay。
7. AI provider 调用。
8. 定时任务执行。

Trace attributes：

| 属性 | 说明 |
| --- | --- |
| `tenant.id` | 租户 |
| `user.id` | 用户 |
| `datasource.key` | 数据源 |
| `ai.provider` | AI provider |
| `ai.model` | 模型 |
| `messaging.topic` | 消息 topic |
| `job.name` | 任务名 |

## 6. 健康检查

| Endpoint | 检查内容 | 说明 |
| --- | --- | --- |
| Liveness | JVM 进程是否健康 | 不检查外部依赖 |
| Readiness | DB、Redis、Kafka、配置中心关键依赖 | 失败则摘流量 |
| Startup | 初始化是否完成 | 防止慢启动被杀 |

Readiness 不应因为非核心依赖失败而全局失败。例如 AI provider 不可用不应导致订单服务完全不可用，除非该服务声明 AI 是核心依赖。

## 7. SLO 默认值

| 服务类型 | 可用性 | P95 延迟 | 错误率 |
| --- | --- | --- | --- |
| 核心交易 API | 99.9% | < 300ms | < 0.1% |
| 普通业务 API | 99.5% | < 800ms | < 0.5% |
| 后台管理 API | 99.0% | < 1500ms | < 1% |
| AI/RAG API | 99.0% | 按场景定义 | < 2% |
| 异步任务 | 成功率 > 99% | 按任务定义 | 死信可控 |

## 8. 告警分级

| 级别 | 示例 | 响应 |
| --- | --- | --- |
| P0 | 核心 API 大面积不可用、数据写入失败 | 立即响应 |
| P1 | 错误率持续升高、Kafka 堆积严重 | 30 分钟内响应 |
| P2 | 单租户异常、慢 SQL 增多 | 工作时间处理 |
| P3 | 容量趋势、依赖版本风险 | 计划处理 |

## 9. 灰度自动分析

Argo Analysis 使用指标：

1. `http_server_requests_seconds` P95/P99。
2. 5xx rate。
3. 核心业务成功率。
4. pod restart count。
5. JVM memory pressure。
6. Outbox backlog。
7. 自定义业务指标。

自动回滚同时执行：

1. Rollout 回滚到上一个 ReplicaSet。
2. 网关权重切回旧服务。
3. 配置中心回滚相关变更。
4. 生成事故或变更记录。

## 10. 备份与灾备

| 对象 | 策略 |
| --- | --- |
| CommonDB | 高频备份，定期恢复演练 |
| BizDB | 按业务等级设置 RPO/RTO |
| Nacos 配置 | Git 主源 + 定期导出 |
| 对象存储 | 版本化 + 生命周期 |
| 日志与 trace | 按合规要求保留 |
| local-libs 私服 | 制品仓库备份 |

## 11. 运维验收

1. 新服务接入后无需手工改代码即可看到日志、指标、trace。
2. 任意一次请求能通过 traceId 查询完整链路。
3. 灰度发布能基于指标自动暂停或回滚。
4. 配置变更能追溯到操作人、审批人和 diff。
5. 每个环境有明确的备份和恢复演练记录。
