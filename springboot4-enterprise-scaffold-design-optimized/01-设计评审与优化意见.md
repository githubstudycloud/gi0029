# 设计评审与优化意见

## 1. 总体评价

原设计覆盖范围完整，已经具备企业级脚手架应有的主要主题：多环境、配置中心、多数据源、AI/RAG、认证鉴权、消息异步、依赖治理、编码规范和可观测性。整体方向正确，尤其是“接口先行、可替换、配置驱动、失败可见、依赖收口”这些原则值得保留。

主要问题是默认方案过多、部分技术组合偏激进，且若直接按原文落地，首版脚手架会承担过高的集成复杂度。优化目标不是削弱能力，而是把“平台核心能力”和“可选增强能力”分层，先做能稳定复用的骨架，再逐步启用高复杂能力。

## 2. 做得好的部分

| 方面 | 评价 |
| --- | --- |
| 文档体系 | 分篇清晰，覆盖从架构到运维的完整生命周期 |
| 原则设定 | 接口化、可替换、配置驱动、依赖治理方向正确 |
| 多环境设计 | local/test/beta/pro 区分明确，符合企业交付流程 |
| 配置治理 | 动态配置、路由配置、静态配置分层是正确方向 |
| 异常规范 | 禁止业务层泛化 `catch Exception`，并保留部分失败模型，工程价值高 |
| 观测基线 | 日志、指标、链路、SLO、灰度回滚有关联，具备可运维性 |

## 3. 主要问题

### 3.1 技术基线需要重新校准

原文将 Spring Boot 4、Spring AI、Spring Cloud Gateway、Nacos、ShardingSphere 等组合直接写为默认链路，但这些组件跨大版本时兼容性变化较大。尤其 Spring Boot 4 进入 Spring Framework 7、Jakarta EE 11、Jackson 3 生态后，第三方 starter 的兼容性不能默认假定。

优化建议：

1. 把 JDK 21 作为运行基线，JDK 25 只进入 CI 验证矩阵。
2. Spring AI 写为“Spring AI 2.0.x 兼容线，GA 后锁定 GA；GA 前仅作为可选 starter 验证”，不要写“Spring AI 4.x”。
3. Nacos/Spring Cloud Alibaba、ShardingSphere、Casbin 等三方集成必须先做 P0 spike；Nacos 需按新版本 `spring.config.import` 模式验证。
4. 所有三方 starter 都通过 `platform-starters` 隔离，不让业务代码直接绑定。

### 3.2 首版范围过大

原文默认包含 Spring Cloud Gateway、Nacos、Argo Rollouts、ShardingSphere、pgvector、Spring AI、Kafka、RocketMQ、XXL-Job、OpenTelemetry 等。如果首版全部并行落地，会导致脚手架发布周期过长，也会让业务团队难以理解边界。

优化建议：

1. v1 只交付核心骨架、网关、配置、认证、单消息中间件、基础观测。
2. 分库分表、CDC、Service Mesh、双 AI 框架、多 MQ 并存作为 v1.1+ 能力。
3. 每个 starter 必须有一个最小可运行 sample，未通过 sample 不进入推荐组合。

### 3.3 模块边界需要更硬

原文提出 SPI 和 starter，但模块结构里容易出现“平台模块互相直连”的风险。若不设 ArchUnit 和 Maven Enforcer 规则，后续会变成大而全公共包。

优化建议：

1. `platform-core` 只放上下文、异常、响应、日志、工具接口，不放业务能力。
2. `platform-starters` 只面向 Spring Boot 自动装配，不被业务领域层直接引用实现类。
3. `platform-integrations` 放三方适配，实现 SPI，不向业务暴露。
4. `samples` 不参与平台依赖反向引用。

### 3.4 配置中心双源设计需要明确主从

原文提出 GitLab + DB + Nacos，但如果没有明确主源，会出现双写冲突、审计缺失和回滚语义不清。

优化建议：

1. Git 是声明式配置主源，适合平台、环境、网关路由模板。
2. Nacos 是运行时分发面，负责动态推送。
3. DB 是业务自助配置和审批工作流存储，不直接作为最终配置事实源。
4. 配置变更必须生成变更单、diff、审批记录和回滚点。

### 3.5 多数据源设计需要降复杂度

原文默认引入 ShardingSphere，并把读写分离、分库分表、影子库都作为近期开口。对脚手架首版来说，这会显著增加兼容性和排障成本。

优化建议：

1. v1 默认支持 CommonDB + BizDB 路由 + 读写分离轻量实现。
2. ShardingSphere 作为 `starter-datasource-shardingsphere` 可选增强。
3. 跨库事务默认禁止；采用 Outbox、Saga 或补偿，而不是默认 XA/Seata。
4. 数据源路由上下文必须可观测，日志与 trace 必须记录 `tenantId`、`dsKey`。

### 3.6 “近 4 年依赖”规则需要例外机制

依赖新鲜度治理有价值，但若一刀切，会误伤稳定协议库、官方 LTS、JDBC 驱动、企业内部 SDK。

优化建议：

1. 官方根 BOM、JDK、Spring、数据库驱动、协议规范库按安全维护状态判断，不只按发布时间判断。
2. 三方依赖设置“建议期、警告期、阻断期”。
3. local-libs 只能作为临时隔离区，必须带 owner、替换计划、到期日期。
4. 禁止 Maven `systemPath` 作为推荐方案，优先同步到私服。

## 4. 优先级建议

| 优先级 | 必做项 | 暂缓项 |
| --- | --- | --- |
| P0 | JDK/Maven/BOM、核心模块、异常响应、日志上下文、配置模型、CI 质量门禁 | 分库分表、AI 深度 RAG、Service Mesh |
| P1 | 网关、认证、Nacos 适配、Kafka Outbox、OpenTelemetry、local/test/beta/pro 模板 | RocketMQ、CDC、SAML 完整后台 |
| P2 | AI starter、pgvector、灰度自动分析、业务自助配置平台 | 多云、多活、多模型复杂调度 |
| P3 | ShardingSphere、Seata/Saga 框架化、Higress/APISIX 适配 | 自研规则引擎 |

## 5. 结论

原设计适合作为能力地图，但不适合直接作为首版实施清单。优化后建议采用“核心平台先稳定、能力 starter 渐进增强”的路线。这样能保留企业级扩展方向，同时把首版交付风险控制在可验证范围内。
