# 依赖治理与编码规范

## 1. 目标

依赖治理和编码规范是脚手架的强制能力。它们决定平台能否长期演进，而不是上线后靠人工约定维护。

## 2. BOM 策略

所有版本只允许在以下位置定义：

1. `platform-bom`：业务可导入的 BOM。
2. `platform-dependencies`：平台内部依赖聚合。
3. Maven pluginManagement：插件版本。

禁止：

1. 子模块直接写依赖版本。
2. 业务项目绕过 BOM 指定平台依赖版本。
3. 同一依赖出现多个版本。

## 3. 依赖准入

依赖进入平台前必须检查：

| 检查 | 工具/方式 |
| --- | --- |
| 版本统一 | Maven Enforcer |
| 漏洞 | OWASP Dependency-Check 或企业 SCA |
| License | License Maven Plugin |
| 新鲜度 | Maven Central/OSS Index/自建索引 |
| 重复类 | duplicate-finder |
| 体积 | dependency tree + image size |
| 传递依赖 | dependency:tree 审查 |

## 4. “近 4 年”规则优化

原规则“超过 4 年必须 local-lib”过于机械。优化为三段式：

| 状态 | 条件 | 动作 |
| --- | --- | --- |
| 正常 | 最近维护或官方 LTS 支持 | 允许 |
| 警告 | 2 年以上无发布但无漏洞 | CI 警告，要求 owner |
| 阻断 | 4 年以上无维护且非豁免 | 禁止引入，需架构审批 |

豁免范围：

1. JDK 和 Spring 官方 BOM 管理的基础依赖。
2. 数据库驱动等供应商仍维护但发布节奏慢的组件。
3. 协议规范类库，且无 CVE、无替代需求。
4. 企业内部 SDK，但必须有 owner 和源码可维护性说明。

## 5. local-libs 策略

local-libs 是隔离区，不是长期仓库。

要求：

1. 禁止推荐 Maven `systemPath`。
2. local jar 必须同步到 Nexus/Artifactory 私服。
3. 必须提供 `META.yaml`。
4. 必须设置 owner、替换计划和到期日期。
5. 每季度清理。

`META.yaml` 示例：

```yaml
gav: com.vendor:legacy-sdk:1.2.3-platform
source: vendor offline package
reason: upstream no longer published to public repository
owner: platform-team
introducedAt: 2026-05-06
expireAt: 2026-11-06
replacementPlan: replace with official rest adapter
securityReview: SEC-2026-001
```

## 6. 工具封装

需要封装：

| 类型 | 策略 |
| --- | --- |
| HTTP | `HttpExecutor` 接口，默认 JDK HttpClient |
| JSON | `JsonCodec` 接口，默认 Jackson 3 |
| 加密 | `CryptoService` 接口，屏蔽具体 provider |
| 时间 | 统一 `Clock` 注入 |
| ID | `IdGenerator` 接口 |
| 对象存储 | `ObjectStorageClient` 接口 |
| AI | `AiClient` 和 `EmbeddingClient` |

不需要过度封装：

1. JDK 基础集合。
2. Spring 标准注解。
3. 简单字符串工具。
4. 测试断言库。

## 7. 编码硬约束

| 规则 | 默认 | 工具 |
| --- | --- | --- |
| 单方法行数 | ≤ 50 | Checkstyle |
| 圈复杂度 | ≤ 4 | PMD |
| 类公开方法数 | ≤ 7 建议，超出警告 | PMD |
| 包深度 | ≤ 6 | ArchUnit |
| Controller 直接访问 Repository | 禁止 | ArchUnit |
| 业务层 `catch Exception` | 禁止 | ArchUnit |
| 子模块直接写版本 | 禁止 | Maven Enforcer |

圈复杂度 4 比较严格，建议新代码强制、存量迁移设置临时基线。

## 8. 异常体系

只允许三类根异常：

| 异常 | 说明 | HTTP |
| --- | --- | --- |
| `BusinessException` | 业务可预期失败 | 400/409/422 |
| `SystemException` | 系统内部失败 | 500 |
| `ThirdPartyException` | 外部系统失败 | 502/503/504 |

业务层允许捕获的异常：

1. 为了转换三方异常，并立即抛出平台异常。
2. 为了部分失败模型收集单项错误。
3. 为了资源关闭或补偿，且不能吞异常。

## 9. 部分失败模型

适用场景：

1. 批量导入。
2. 批量通知。
3. 多 provider fallback。
4. 多文件处理。

模型：

```java
public record PartialResult<T>(
    List<T> successes,
    List<PartialFailure> failures,
    PartialPolicy policy
) {}
```

策略：

| 策略 | 说明 |
| --- | --- |
| `FAIL_FAST` | 任一失败整体失败 |
| `BEST_EFFORT` | 尽力处理，返回部分失败 |
| `MIN_SUCCESS_RATIO` | 达到成功比例才算成功 |

## 10. 日志规范

1. 日志必须结构化输出 JSON。
2. 必须包含 traceId、requestId、tenantId、userId、app、env。
3. 禁止打印密码、token、身份证、银行卡等敏感信息。
4. 异常日志只在边界层打一次。
5. 业务失败使用 warn，系统异常使用 error。

## 11. 测试规范

| 类型 | 要求 |
| --- | --- |
| 单元测试 | domain 和 usecase 必须覆盖核心分支 |
| 集成测试 | 数据库、Redis、Kafka 使用 Testcontainers |
| 契约测试 | OpenAPI 和消息 schema 必须校验 |
| 架构测试 | ArchUnit 进入 `mvn verify` |
| 回归测试 | sample 应用必须全部启动通过 |
