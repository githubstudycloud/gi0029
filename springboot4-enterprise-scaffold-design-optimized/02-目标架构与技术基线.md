# 目标架构与技术基线

## 1. 架构目标

目标是构建一个能被多个业务团队复用的 Spring Boot 4 企业级脚手架，提供统一工程骨架、统一依赖治理、统一异常响应、统一认证网关、统一配置治理、统一观测基线，并通过 starter 机制提供 AI、多数据源、消息、任务调度等可选能力。

关键不是“把所有能力做进核心”，而是让业务团队能在同一套治理标准下选择能力模块。

## 3. 技术基线

| 类别 | 基线 | 说明 |
| --- | --- | --- |
| Java | JDK 21 LTS | 运行和编译默认基线 |
| Spring Boot | 4.x | 统一使用 Spring Boot 4 系列 |
| Spring Framework | 7.x | 由 Boot 4 管理，不单独指定 |
| Jakarta EE | 11 | 所有 `javax.*` 迁移到 `jakarta.*` |
| JSON | Jackson 3 | 禁止业务直接依赖旧 Jackson 2 API 细节 |
| 构建 | Maven 3.9+ + Wrapper | 根 POM 管理所有模块 |
| 测试 | JUnit 5 + Testcontainers | 集成测试依赖容器化 |
| 质量 | Checkstyle + PMD + SpotBugs + ArchUnit + Maven Enforcer | CI 强制 |
| 观测 | OpenTelemetry + Micrometer | 自动注入 traceId/requestId/tenantId |

## 4. 三方组件默认策略

| 领域 | 默认 | 可选 | 首版策略 |
| --- | --- | --- | --- |
| 配置中心 | Nacos | Spring Cloud Config, Apollo | Nacos 先做兼容性 spike |
| 网关 | Spring Cloud Gateway | Higress, APISIX | 只实现 Gateway |
| 数据库 | PostgreSQL | MySQL | 双方言测试，但示例默认 PG |
| 缓存 | Redis | Caffeine | Redis 作为分布式缓存，Caffeine 本地缓存 |
| 消息 | Kafka | RocketMQ, RabbitMQ | v1 只默认 Kafka |
| 调度 | XXL-Job | Quartz, K8s CronJob | 任务 starter 可选 |
| AI | Spring AI 2.0.x 兼容线 | LangChain4j | v1 只做 Spring AI 薄封装；GA 前保持可选 |
| 向量库 | pgvector | Milvus, Elasticsearch Vector | 起步使用 pgvector |
| 鉴权 | Spring Security OAuth2 Resource Server | Keycloak, Auth0, Azure AD, 企业微信/钉钉 | OIDC 优先 |

## 5. 必须验证的兼容性

| 编号 | 验证项 | 通过标准 |
| --- | --- | --- |
| S-1 | Spring Cloud Gateway 与 Spring Boot 4 | 网关启动、路由、过滤器、限流、OIDC 资源服务可用 |
| S-2 | Nacos/Spring Cloud Alibaba 与 Spring Boot 4 | 基于 `spring.config.import` 的配置加载、动态刷新、namespace/group、灰度配置可用 |
| S-3 | MyBatis 系与 Boot 4/Jakarta | mapper 扫描、事务、分页、审计字段可用 |
| S-4 | Kafka client 与 Boot 4 | 生产、消费、重试、死信、事务外 Outbox 可用 |
| S-5 | Spring AI 2.0.x 与 Boot 4/Jackson 3 | chat、embedding、stream、tool calling、vector store 可用 |
| S-6 | OpenTelemetry 自动埋点 | HTTP、JDBC、Redis、Kafka、RestClient trace 串联 |

## 6. 不进入首版默认组合的能力

| 能力 | 原因 | 后续入口 |
| --- | --- | --- |
| Service Mesh/Istio | 运维复杂度高，首版不需要 | v2 多服务治理阶段 |
| ShardingSphere 默认启用 | Boot 4 兼容性和排障成本高 | 可选 starter |
| Seata/XA | 跨库事务会放大复杂度 | 明确强一致场景再启用 |
| 双 AI 框架 | 抽象成本高，接口不稳定 | Provider SPI 预留 |
| 多 MQ 同时默认支持 | 契约和运维差异大 | 先 Kafka，后 RocketMQ |
| CDC/Debezium | 基础设施重 | Outbox 稳定后再扩展 |

## 7. 设计约束

1. 业务模块不得直接依赖三方 SDK，必须通过平台接口或 adapter。
2. 所有依赖版本只允许在 BOM 中定义。
3. 所有 starter 必须提供自动配置、配置元数据、健康检查和最小 sample。
4. 所有跨服务交互必须使用 OpenAPI、消息契约或明确的 SPI。
5. 所有请求链路必须带 `requestId`、`traceId`、`tenantId`、`userId` 中可获取的字段。
