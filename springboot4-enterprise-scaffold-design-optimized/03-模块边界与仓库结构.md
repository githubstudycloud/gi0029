# 模块边界与仓库结构

## 1. 仓库形态

采用“脚手架 mono-repo + 业务独立仓”的组合：

1. 脚手架仓库负责平台能力、BOM、starter、CI 模板、示例应用和 archetype。
2. 业务仓库通过 BOM 和 starter 依赖平台能力，不复制平台源码。
3. 业务团队可以从 archetype 生成工程，也可以 fork sample 作为起点。

这种方式兼顾集中治理和业务独立演进，避免所有业务代码进入一个巨大 mono-repo。

## 3. 模块职责

| 模块 | 职责 | 禁止事项 |
| --- | --- | --- |
| `platform-bom` | 统一依赖版本 | 写业务逻辑 |
| `platform-core` | 上下文、异常、响应、日志、校验、测试基类 | 依赖 Nacos/Kafka/AI 等三方实现 |
| `platform-contracts` | OpenAPI、消息 schema、SPI 接口 | 放实现类 |
| `platform-starters` | Spring Boot 自动装配和配置属性 | 让业务直接调用内部实现 |
| `platform-integrations` | 三方 SDK 适配 | 反向依赖业务模块 |
| `platform-governance` | 质量规则和 CI 门禁 | 混入运行时代码 |
| `samples` | 验证 starter 可用性 | 被平台模块依赖 |

## 4. 依赖方向

允许方向：

```text
samples
  └── platform-starters
        ├── platform-core
        ├── platform-contracts
        └── platform-integrations
              ├── platform-core
              └── platform-contracts
```

禁止方向：

1. `platform-core` 依赖 `platform-starters`。
2. `platform-core` 依赖具体三方集成。
3. `platform-integrations` 依赖 `samples` 或业务模块。
4. 业务模块依赖另一个业务模块的 Service 实现类。
5. 跨服务直接访问对方业务库。

## 6. 分层规则

| 层 | 允许依赖 | 说明 |
| --- | --- | --- |
| `interfaces` | `application`, DTO, core-web | REST、消息入口、参数转换 |
| `application` | `domain`, contracts, core-error | Use Case 编排、事务边界 |
| `domain` | core 基础类型 | 业务规则，不感知 Spring 和数据库 |
| `infrastructure` | domain repository 接口、integrations | 数据库、外部系统、防腐层 |

强制规则：

1. Controller 不直接调用 Repository。
2. Domain 不依赖 Spring Web、MyBatis、Kafka、Redis。
3. Adapter 捕获三方异常并翻译为平台异常。
4. Use Case 是事务入口，复杂业务不得堆在 Controller。

## 7. Starter 标准

每个 starter 必须包含：

1. `@AutoConfiguration` 自动装配。
2. `@ConfigurationProperties` 配置属性。
3. `additional-spring-configuration-metadata.json` 元数据。
4. HealthIndicator 或观测指标。
5. 最小 sample。
6. README，说明依赖、配置、启停方式、常见问题。

## 8. 业务接入流程

1. 从 archetype 创建业务工程。
2. 引入平台 BOM。
3. 选择需要的 starter。
4. 配置应用名、环境、租户策略、数据源策略。
5. 执行本地 `docker compose up`。
6. 运行 `mvn verify`，通过质量门禁。
7. 接入 CI 模板和 Helm values。
